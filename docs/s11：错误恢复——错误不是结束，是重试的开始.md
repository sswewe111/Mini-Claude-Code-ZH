# s11：错误恢复——错误不是结束，是重试的开始

Error Recovery 的目标是让 Agent 在模型调用失败时先尝试恢复，而不是直接让程序崩溃。

当前项目把错误恢复集中放在 `RecoveryManager` 中。主 Agent 和子 Agent 都通过它调用模型；它负责错误分类、临时错误重试、上下文超限恢复和输出截断续写。

---

## 为什么需要 Error Recovery

Agent 运行时常见的失败不一定是代码错误：

- 模型服务临时返回 `502`、`503`、`529`。
- 请求过快触发 `429 rate limit`。
- 上下文太长，服务端拒绝请求。
- 模型回答太长，返回 `finish_reason == "length"`。
- 网络超时或连接被重置。

如果这些情况直接抛异常，Agent Loop 就会中断。Error Recovery 的作用是在这些错误发生时根据类型选择恢复策略。

## 配置文件

`configs/recovery_config.yml` 控制错误恢复策略：

```yaml
ENABLE_RECOVERY: true
MAX_TRANSIENT_RETRIES: 3
MAX_CONTEXT_RECOVERY_RETRIES: 1
MAX_CONTINUATION_RETRIES: 2
BACKOFF_BASE_SECONDS: 1
BACKOFF_MAX_SECONDS: 20
BACKOFF_JITTER_RATIO: 0.25
FALLBACK_AFTER_OVERLOADS: 3
FALLBACK_MODEL_ID: ""
```

关键配置含义：

- `ENABLE_RECOVERY`：是否启用错误恢复。
- `MAX_TRANSIENT_RETRIES`：临时错误最大重试次数。
- `MAX_CONTEXT_RECOVERY_RETRIES`：上下文超限后 reactive compact 的最大次数。
- `MAX_CONTINUATION_RETRIES`：输出截断后最多续写几轮。
- `RETRY_STATUS_CODES`：可重试状态码，例如 `429/500/502/503/504/529`。
- `CONTEXT_ERROR_KEYWORDS`：用于识别上下文超限错误。
- `TRANSIENT_ERROR_KEYWORDS`：用于识别超时、连接错误、服务过载等临时错误。
- `FALLBACK_MODEL_ID`：备用模型；为空时不会切换模型。

配置通过 `utils/config_handler.py` 中的 `load_recovery_config()` 加载，并导出为 `recovery_config`。

## RecoveryManager 的职责

`managers/recovery_manager.py` 是错误恢复的核心。

它不负责执行工具，不负责权限控制，也不负责真正压缩上下文。它只负责判断模型调用失败后应该怎么恢复。

主要方法：

```python
create_chat_completion(...)
```

带恢复能力地调用 `client.chat.completions.create(...)`。

它会：

- 每次重试前重新执行 `build_messages()`。
- 临时错误时指数退避后重试。
- 上下文超限时调用 `CompactManager.reactive_compact(...)` 后重试。
- 成功后返回模型 response。

```python
build_continuation_message(finish_reason)
```

当 `finish_reason == "length"` 时生成中文续写消息。

```python
classify_error(exc)
```

把不同 OpenAI 兼容服务的异常统一分类为：

- `context_length`
- `rate_limit`
- `server_overloaded`
- `network_timeout`
- `transient`
- `unknown`

## 状态记录

`state/recovery_state.py` 中的 `RecoveryState` 保存一次 Agent 运行中的恢复状态：

```python
transient_retries
context_retries
continuation_retries
consecutive_overloads
switched_fallback_model
current_model_id
last_error_type
last_error_reason
```

这些状态用于避免无限重试。例如临时错误超过 `MAX_TRANSIENT_RETRIES` 后会停止恢复并抛出错误。

## 主 Agent 如何接入

主 Agent 在 `agent_loop.py` 中创建 `RecoveryManager`：

```python
recovery_manager = RecoveryManager()
```

模型调用不再直接写成：

```python
client.chat.completions.create(...)
```

而是改成：

```python
response = recovery_manager.create_chat_completion(
    client=client,
    model_id=model_id,
    build_messages=build_messages,
    tools=BASE_TOOLS,
    tool_choice="auto",
    max_tokens=_max_output_tokens(),
    compact_manager=COMPACT_MANAGER,
    state_messages=state.messages,
)
```

这里传入的是 `build_messages` 函数，而不是固定的 `messages` 列表。原因是：如果发生上下文超限，`RecoveryManager` 会调用 `reactive_compact` 修改 `state.messages`，下一次重试需要重新构建最新的模型输入。

模型返回后，主循环检查是否被截断：

```python
continuation_message = recovery_manager.build_continuation_message(finish_reason)
if continuation_message:
    state.messages.append(continuation_message)
    continue
```

如果 `finish_reason == "length"`，就追加一条中文续写 user message，然后进入下一轮循环。

## 子 Agent 如何接入

子 Agent 在 `subagents/task_subagent.py` 中也创建自己的 `RecoveryManager`：

```python
recovery_manager = RecoveryManager()
```

子 Agent 调用模型时同样使用：

```python
response = recovery_manager.create_chat_completion(...)
```

子 Agent 的错误恢复只影响子 Agent 自己的 `state.messages`：

- 子 Agent 上下文超限时，只压缩子 Agent 的上下文。
- 子 Agent 输出截断时，只给子 Agent 追加续写提示。
- 子 Agent 临时 API 错误时，自己退避重试。

它不会直接修改主 Agent 的历史消息。

## 恢复流程

### 1. 正常调用前

Agent Loop 先执行已有的 `BeforeModelCall` hooks：

- Todo reminder。
- Memory 注入。
- Context Compact 的调用前压缩。
- System Prompt 运行时组装。

这些属于正常调用前准备，不属于错误恢复。

### 2. 模型调用阶段

`RecoveryManager.create_chat_completion(...)` 调用模型。

如果调用成功，直接返回 response。

如果调用失败，先用 `classify_error(exc)` 分类。

### 3. 临时错误恢复

触发条件：

- `429`
- `500`
- `502`
- `503`
- `504`
- `529`
- timeout
- connection error
- overloaded

恢复动作：

1. 记录错误类型。
2. 计算退避时间。
3. `time.sleep(delay)`。
4. 重试同一个请求。

退避公式：

```python
delay = min(base * 2 ** (attempt - 1), max_delay) + jitter
```

如果服务端返回 `Retry-After`，优先使用服务端建议的等待时间。

### 4. 上下文超限恢复

触发条件：

异常文本命中 `CONTEXT_ERROR_KEYWORDS`，例如：

- `context length`
- `prompt too long`
- `too many tokens`
- `context_length_exceeded`
- `413`

恢复动作：

1. 调用 `CompactManager.reactive_compact(...)`。
2. reactive compact 会把当前历史压缩成摘要。
3. `RecoveryManager` 重新执行 `build_messages()`。
4. 使用压缩后的上下文重试模型调用。

注意：s11 不重新实现压缩。真正的压缩逻辑仍然属于 s08 的 `CompactManager`。

### 5. 输出截断恢复

触发条件：

模型调用成功，但返回：

```python
finish_reason == "length"
```

恢复动作：

1. 当前 assistant 消息先写入 `state.messages`。
2. `RecoveryManager.build_continuation_message(...)` 生成中文续写提示。
3. 把续写提示作为 user message 追加进 `state.messages`。
4. `continue` 进入下一轮 Agent Loop。

超过 `MAX_CONTINUATION_RETRIES` 后不再续写，避免无限循环。

---

## 与其他模块的边界

### 与 Context Compact 的关系

Error Recovery 只判断什么时候需要应急压缩。

真正压缩由：

```python
CompactManager.reactive_compact(...)
```

完成。

### 与 Hook 的关系

当前实现没有新增 `ModelError` hook。

原因是错误恢复属于模型调用控制流，放在 `RecoveryManager` 中更直接。Hook 仍然负责 `BeforeModelCall`、`AfterModelCall`、`PreToolUse`、`PostToolUse` 等横切逻辑。

### 与工具错误的关系

工具错误不由 `RecoveryManager` 自动重试。

工具参数错误、权限拒绝、工具执行异常仍然由 dispatcher 返回 tool result，让模型在下一轮自行修正。

---

## 当前效果

现在主 Agent 和子 Agent 都具备以下能力：

- 临时 API 错误不会立刻中断。
- 上下文超限时会触发 reactive compact 后重试。
- 输出被截断时会自动追加中文续写提示。
- 重试次数受配置限制，不会无限循环。
- 失败请求不会写入 `state.messages`，避免污染上下文。

---

## 测试建议

可以通过 fake client 或临时 mock 来测试：

- 前两次抛出 `502`，第三次返回成功，检查是否重试 3 次以内成功。
- 抛出包含 `prompt too long` 的异常，检查是否调用 reactive compact。
- 构造 `finish_reason == "length"` 的 response，检查是否追加中文续写消息。
- 把 `MAX_TRANSIENT_RETRIES` 设为 `1`，连续抛出 `502`，检查是否停止重试并抛出错误。
