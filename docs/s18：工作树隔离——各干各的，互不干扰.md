# s18：工作树隔离——各干各的，互不干扰

[中文](README.md) | [English](README.en.md) | [日本語](README.ja.md)

s01 → ... → s16 → s17 → `s18` → s19 → s20

> “各干各的，互不干扰” 的目标是：任务系统负责“做什么”，Autonomous Agents 负责“谁来做”，Worktree Isolation 负责“在哪做”。
>
> Harness 层：隔离执行。Lead 可以给任务创建独立 git worktree；teammate 自动认领任务后，在绑定的隔离目录里执行文件和命令工具。

## 解决的问题

s17 让 teammate 可以自己看板、自己认领，但多个 teammate 仍共享同一个项目工作目录。并行写代码时会有几个问题：

- 两个 teammate 可能修改同一个文件，互相覆盖。
- 很难分清某个任务产生了哪些改动。
- 想丢弃某个 teammate 的实验性改动时，缺少单独的隔离边界。

s18 使用 git worktree 解决“在哪做”的问题：

1. Lead 为任务创建独立 worktree，例如 `.worktrees/auth-refactor`。
2. worktree 使用独立分支，例如 `wt/auth-refactor`。
3. Lead 把 worktree 绑定到任务的 `metadata.worktree`。
4. teammate 自动认领任务后，如果任务绑定了 worktree，就把自己的工具执行目录切到该 worktree。
5. `bash/read_file/write_file/edit_file` 都在 teammate 的局部 `current_workdir` 下运行。

这样两个 teammate 即使写同名文件，也会写到不同 worktree 目录里。

## 配置项

当前配置文件为 `configs/worktree_config.yml`：

```yaml
ENABLE_WORKTREE_ISOLATION: true
WORKTREE_ROOT: ".worktrees"
WORKTREE_BRANCH_PREFIX: "wt/"
WORKTREE_INDEX_FILE: ".worktrees/index.json"
WORKTREE_EVENT_FILE: ".worktrees/events.jsonl"
WORKTREE_NAME_PATTERN: "^[A-Za-z0-9._-]{1,64}$"
WORKTREE_BASE_REF: "HEAD"
WORKTREE_ALLOW_CREATE_WITH_DIRTY_REPO: true
WORKTREE_REMOVE_REQUIRES_CLEAN: true
WORKTREE_BIND_METADATA_KEY: "worktree"
WORKTREE_AUTO_ENTER_ON_CLAIM: true
WORKTREE_NOTIFY_LEAD_ON_BIND: true
```

| 配置 | 当前值 | 说明 |
|---|---:|---|
| `ENABLE_WORKTREE_ISOLATION` | `true` | 总开关 |
| `WORKTREE_ROOT` | `.worktrees` | worktree 根目录 |
| `WORKTREE_BRANCH_PREFIX` | `wt/` | 自动创建分支名前缀 |
| `WORKTREE_INDEX_FILE` | `.worktrees/index.json` | worktree 索引文件 |
| `WORKTREE_EVENT_FILE` | `.worktrees/events.jsonl` | worktree 事件流 |
| `WORKTREE_NAME_PATTERN` | `^[A-Za-z0-9._-]{1,64}$` | worktree 名称白名单 |
| `WORKTREE_BASE_REF` | `HEAD` | 默认基于哪个 ref 创建 worktree |
| `WORKTREE_ALLOW_CREATE_WITH_DIRTY_REPO` | `true` | 主工作区有改动时仍允许创建 worktree |
| `WORKTREE_REMOVE_REQUIRES_CLEAN` | `true` | 删除前要求 worktree 干净 |
| `WORKTREE_BIND_METADATA_KEY` | `worktree` | 写入任务 metadata 的 key |
| `WORKTREE_AUTO_ENTER_ON_CLAIM` | `true` | teammate 认领绑定任务后自动使用隔离目录 |

`worktree_remove` 已加入 `configs/permission_config.yml` 的 `required_tools`，删除 worktree 会走权限审批。

## WorktreeRecord

`state/worktree_state.py` 定义运行态记录：

```python
@dataclass
class WorktreeRecord:
    name: str
    path: str
    branch: str
    base_ref: str = "HEAD"
    task_id: str = ""
    status: str = "active"
    created_at: str = ""
    updated_at: str = ""
    metadata: Dict = field(default_factory=dict)
```

索引文件 `.worktrees/index.json` 的形态：

```json
{
  "worktrees": [
    {
      "name": "auth-refactor",
      "path": "D:/.../.worktrees/auth-refactor",
      "branch": "wt/auth-refactor",
      "base_ref": "HEAD",
      "task_id": "task_000021",
      "status": "active",
      "created_at": "...",
      "updated_at": "...",
      "metadata": {}
    }
  ]
}
```

状态当前主要有：

- `active`：可被任务使用。
- `kept`：保留用于 review。
- `removed`：已删除。

## Lead 工具

当前 Lead 可用的 worktree 工具来自 `tools_configs/worktree_configs.py`：

| 工具 | 作用 |
|---|---|
| `worktree_create` | 创建 git worktree，可选绑定 `task_id` |
| `worktree_bind` | 把已有 worktree 绑定到任务 metadata |
| `worktree_list` | 列出 worktree 记录 |
| `worktree_status` | 查看某个 worktree 的 git status、分支、路径和绑定任务 |
| `worktree_keep` | 标记为保留，用于人工 review |

1. `tools_configs/worktree_configs.py` 定义 `WORKTREE_TOOLS`。
2. `tools_configs/base_configs.py` 把 `WORKTREE_TOOLS` 合并进 `BASE_TOOLS`。

4. `handlers/base_handlers.py` 将 `WORKTREE_HANDLERS` 合并进 `BASE_HANDLERS`。
5. `configs/permission_config.yml` 将 `worktree_*` 工具加入 allow 列表。

## 创建和绑定流程

创建入口是 `WorktreeManager.create(name, task_id="", base_ref="")`。

流程：

1. 检查 `ENABLE_WORKTREE_ISOLATION`。
2. 校验 worktree 名称：
   - 不能是空、`.`、`..`。
   - 不能包含 `/` 或 `\`。
   - 必须匹配 `WORKTREE_NAME_PATTERN`。
3. 计算目录和分支：
   - path：`.worktrees/<name>`
   - branch：`wt/<name>`
4. 用 `git rev-parse --show-toplevel` 确认当前目录是 git 仓库。
5. 如果配置禁止 dirty repo，则检查 `git status --porcelain`。
6. 执行：

```text
git worktree add -b wt/<name> .worktrees/<name> HEAD
```

7. 写入 `.worktrees/index.json`。
8. 写入 `.worktrees/events.jsonl` 的 `create` 事件。
9. 如果传入 `task_id`，继续调用 `bind_task()`。

绑定入口是 `WorktreeManager.bind_task(task_id, name)`。

绑定只写任务 metadata，不认领任务，不改变任务状态：

```json
{
  "metadata": {
    "worktree": {
      "name": "auth-refactor",
      "path": ".worktrees/auth-refactor",
      "branch": "wt/auth-refactor"
    }
  }
}
```

这是 s18 的关键边界：任务状态仍由 `TaskManager.claim()` 和 `TaskManager.complete()` 推进，worktree 只提供执行目录隔离。

## teammate 如何自动进入隔离目录

s17 的自动认领发生在 `subagents/teammate_agent.py` 的 `_idle_until_message()` 中。

s18 在认领成功后追加了 worktree 处理：

1. `AUTONOMOUS_MANAGER.try_claim_next()` 成功返回 `claim_result.task`。
2. `WORKTREE_MANAGER.format_worktree_context(task_id)` 读取任务绑定信息，并把 worktree 信息追加到 `<claimed_task>` 后面。
3. `WORKTREE_MANAGER.resolve_task_workdir(task_id)` 返回真实 worktree 目录。
4. 如果存在 worktree：
   - `tool_context["current_workdir"] = workdir`
   - `tool_context["current_task_id"] = task_id`
   - 写入 `.worktrees/events.jsonl` 的 `auto_enter` 事件
5. 如果不存在 worktree，则回到原始项目工作区。

注入给 teammate 的内容会包含类似：

```text
worktree:
name: auth-refactor
path: .worktrees/auth-refactor
branch: wt/auth-refactor
你的文件和命令工具会在该隔离目录中执行。
```

teammate 不需要手动调用 `worktree_bind` 或 `worktree_create`。

## 为什么不用 os.chdir

当前项目的 teammate 是多线程运行的。如果调用全局 `os.chdir()`，会影响同一进程里的所有 teammate 和主 Agent。

所以 s18 使用局部上下文：

```python
tool_context = {
    "original_workdir": workdir,
    "current_workdir": workdir,
    "current_task_id": "",
}
```

teammate 调用工具时，把 `current_workdir` 传给工具：

```python
run_bash(args["command"], cwd=current_workdir)
read_file(args["path"], limit, base_dir=current_workdir)
write_file(args["path"], content, base_dir=current_workdir)
edit_file(args["path"], old_text, new_text, base_dir=current_workdir)
```

主 Agent 的 `bash/read_file/write_file/edit_file` 默认行为不变，仍使用主工作区。

## 路径沙箱

s18 增加了 `safe_path_under(root, path)`：

```python
def safe_path_under(root: Path, path: str) -> Path:
    root = Path(root).resolve()
    resolved = (root / path).resolve()
    resolved.relative_to(root)
    return resolved
```

它用于 teammate 在 worktree 中读写文件时的路径校验，防止通过 `..` 跳出隔离目录。

当前路径边界：

- 主 Agent：仍由 `safe_path()` 限制在主 `WORKDIR`。
- teammate 未认领 worktree 任务：使用原始 `workdir`。
- teammate 认领绑定 worktree 的任务：使用 `safe_path_under(worktree_path, path)`。

## keep 和 remove

`worktree_keep(name, reason="")`：

- 将索引状态改为 `kept`。
- 不删除目录。
- 不删除分支。
- 写入 `keep` 事件。
- 返回 path 和 branch，方便人工 review。

`worktree_remove(name, discard_changes=false)`：

1. 读取 worktree 记录。
2. 如果 `WORKTREE_REMOVE_REQUIRES_CLEAN=true` 且 `discard_changes=false`：
   - 执行 `git -C <path> status --porcelain`。
   - 如果有输出，拒绝删除并写 `remove_rejected`。
3. 执行：

```text
git worktree remove <path> --force
git branch -D <branch>
```

4. 将索引状态改为 `removed`。
5. 写入 `remove` 事件。

删除是破坏性操作，所以工具权限配置里 `worktree_remove` 需要审批。

## 事件流

`.worktrees/events.jsonl` 会记录 worktree 生命周期事件。

常见事件：

| 事件 | 触发时机 |
|---|---|
| `create` | `worktree_create` 成功后 |
| `bind_task` | worktree 绑定到 task metadata 后 |
| `auto_enter` | teammate 自动认领绑定 worktree 的任务后 |
| `keep` | `worktree_keep` 后 |
| `remove_rejected` | 删除被拒绝，例如 worktree 有未提交改动 |
| `remove` | `worktree_remove` 成功后 |
| `git_error` | git 命令失败 |

排查时建议同时看：

- `.worktrees/events.jsonl`：worktree 生命周期。
- `.worktrees/index.json`：当前记录和状态。
- `.tasks/events.jsonl`：任务 create/claim/complete。
- `.team/autonomous_events.jsonl`：自治扫描、认领和完成。
- `.team/events.jsonl`：队友消息。

## 示例流程

用户请求：

> 把两个并行修改任务放到隔离目录里，让两个 teammate 各自处理。

理想流程：

1. Lead 创建两个任务：
   - `task_000021`：修改认证模块。
   - `task_000022`：修改登录页面。
2. Lead 创建并绑定 worktree：

```text
worktree_create(name="auth-refactor", task_id="task_000021")
worktree_create(name="ui-login", task_id="task_000022")
```

3. Lead 调用 `task_get` 确认两个任务都有 `metadata.worktree`。
4. Lead 启动 teammate：

```text
spawn_teammate(name="alice", role="backend engineer", prompt="进入 idle 后认领 ready 后端任务...")
spawn_teammate(name="bob", role="frontend engineer", prompt="进入 idle 后认领 ready 前端任务...")
```

5. alice 自动认领 `task_000021`，进入 `.worktrees/auth-refactor`。
6. bob 自动认领 `task_000022`，进入 `.worktrees/ui-login`。
7. 两个 teammate 各自在隔离目录里运行 `bash/read_file/write_file/edit_file`。
8. teammate 调用 `task_complete` 完成任务并向 Lead 汇报。
9. Lead 调用：

```text
worktree_status(name="auth-refactor")
worktree_status(name="ui-login")
```

10. Review 后选择：

```text
worktree_keep(name="auth-refactor", reason="保留给人工 review")
worktree_remove(name="ui-login", discard_changes=false)
```

## 测试建议

可以用下面的测试方向验证 s18：

1. 调用 `worktree_list`，确认初始索引。
2. 用 `task_create` 创建两个 pending 任务。
3. 对每个任务调用 `worktree_create(name=..., task_id=...)`。
4. 调用 `task_get` 确认 `metadata.worktree` 已写入，任务仍是 `pending`。
5. 启动两个 teammate，让它们 idle 后自动认领任务。
6. 要求 teammate 在任务里写不同文件。
7. 检查主工作区没有出现这些测试文件。
8. 检查 `.worktrees/<name>/...` 中出现对应文件。
9. 检查 `.worktrees/events.jsonl` 中有 `create`、`bind_task`、`auto_enter`。
10. 检查 `worktree_status` 输出各自分支和改动。

如果当前目录不是有效 git 仓库，`worktree_create` 会直接返回：

```text
当前工作区不是 git 仓库，无法创建 worktree。
```

这是当前代码的预期行为，不会写入成功记录，也不会绑定任务。

## 相对 s17 的变化

| 维度 | s17 Autonomous Agents | s18 Worktree Isolation |
|---|---|---|
| 工作来源 | teammate idle 后自动认领 ready task | 沿用 s17 |
| 任务状态 | `pending -> in_progress -> completed` | 沿用 `TaskManager`，worktree 不改任务状态 |
| 执行目录 | 默认主 `WORKDIR` | 任务可绑定独立 git worktree |
| 绑定位置 | 无 | `task.metadata.worktree` |
| Lead 新工具 | 无 | `worktree_create/bind/list/status/keep/remove` |
| teammate 新工具 | 无 | 无；通过认领任务后自动切换局部 `current_workdir` |
| 路径沙箱 | 主工作区根目录 | 支持主工作区根目录和 worktree 根目录 |
| 审计文件 | `.team`、`.tasks` | 增加 `.worktrees/index.json` 和 `.worktrees/events.jsonl` |
| 删除策略 | 不涉及 | dirty worktree 默认拒绝删除，`worktree_remove` 需要审批 |
