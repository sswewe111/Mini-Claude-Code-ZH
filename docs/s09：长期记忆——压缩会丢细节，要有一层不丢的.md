# s09：长期记忆——压缩会丢细节，要有一层不丢的

s08 解决的是当前会话上下文过长的问题，但压缩摘要本身是有损的；重启会话后，摘要也不会天然存在。s09 增加长期 Memory，用文件系统保存跨会话仍然有价值的信息，例如用户偏好、协作反馈、项目背景和外部资源入口。

当前项目的 Memory 采用两层加载方式：

- `MEMORY.md` 索引进入 system prompt。
- 相关记忆正文通过 LLM side-query 按需选择，再临时注入当前 messages。

## 文件结构

```text
configs/
└── memory_config.yml
state/
└── memory_state.py
managers/
└── memory_manager.py
prompts/
└── memory_prompts.py
tools_configs/
└── base_configs.py
handlers/
└── base_handlers.py
hooks/
├── builtin_hooks.py
└── hook_manager.py
.memory/
├── private/
│   ├── MEMORY.md
│   └── *.md
└── team/
    ├── MEMORY.md
    └── *.md
```

职责划分：

- `configs/memory_config.yml`：Memory 开关、目录、召回数量、正文预算、自动提取和 Dream 门控。
- `state/memory_state.py`：Memory 元数据和运行时状态。
- `managers/memory_manager.py`：扫描、索引、召回、保存、删除、自动提取和 Dream。
- `prompts/memory_prompts.py`：Memory 使用规则、召回选择 prompt、自动提取 prompt。
- `tools_configs/base_configs.py`：定义 `save_memory`、`forget_memory` 工具 schema。
- `handlers/base_handlers.py`：把 Memory 工具映射到 `MEMORY_MANAGER`。
- `hooks/builtin_hooks.py`：把 Memory 挂到 `SessionStart`、`BeforeModelCall`、`Stop`。

## 配置

当前 `configs/memory_config.yml`：

```yaml
ENABLE_MEMORY: true
PRIVATE_MEMORY_DIR: ".memory/private"
TEAM_MEMORY_DIR: ".memory/team"
MAX_INDEX_LINES: 200
MAX_MEMORY_FILES: 200
MAX_RELEVANT_MEMORIES: 5
MAX_MEMORY_FILE_CHARS: 4096
MAX_TOTAL_MEMORY_CHARS: 60000
AUTO_EXTRACT_MEMORY: true
EXTRACT_RECENT_MESSAGES: 12
MEMORY_RELEVANCE_MAX_TOKENS: 300
MEMORY_EXTRACT_MAX_TOKENS: 1200
DREAM_ENABLED: true
DREAM_MIN_FILES: 10
DREAM_MIN_INTERVAL_HOURS: 24
MEMORY_LOCK_TTL_MINUTES: 60
ALLOW_SUBAGENT_MEMORY_READ: true
ALLOW_SUBAGENT_MEMORY_WRITE: false
```

说明：

- `MAX_RELEVANT_MEMORIES` 控制每轮最多注入几条完整记忆。
- `MEMORY_RELEVANCE_MAX_TOKENS` 控制 LLM side-query 的输出预算。
- `MAX_MEMORY_FILE_CHARS` 控制单个记忆文件注入上限。
- `MAX_TOTAL_MEMORY_CHARS` 控制单轮所有 Memory context 的总预算。
- `ALLOW_SUBAGENT_MEMORY_READ` 默认允许子 Agent 读取 Memory。
- `ALLOW_SUBAGENT_MEMORY_WRITE` 默认禁止子 Agent 写 Memory。

## 记忆存储

Memory 分为 private 和 team 两个作用域：

```text
.memory/private/  # 当前用户私有记忆
.memory/team/     # 项目共享记忆
```

每个作用域都有自己的 `MEMORY.md`：

```text
.memory/private/MEMORY.md
.memory/team/MEMORY.md
```

`MEMORY.md` 只保存索引，不保存完整正文。例如：

```markdown
# Memory

- [user-is-article-reader](user-is-article-reader.md) - 用户是一位文章阅读人员，喜欢阅读文章并总结内容
```

具体记忆是独立 Markdown 文件，使用 YAML frontmatter：

```markdown
---
name: user-is-article-reader
description: 用户是一位文章阅读人员，喜欢阅读文章并总结内容
type: user
scope: private
updated_at: 2026-07-05T14:25:30+08:00
---

## 用户身份与偏好

- **身份**：文章阅读人员
- **偏好**：喜欢阅读文章并进行内容总结
```

## 记忆类型

当前支持四类：

```text
user:
  scope: 必须 private
  内容: 用户身份、背景、长期偏好、协作方式

feedback:
  scope: 默认 private；明确是项目规则时用 team
  内容: 用户对 Agent 工作方式的纠正或确认

project:
  scope: private 或 team，通常偏向 team
  内容: 项目背景、目标、决策、约束、绝对日期

reference:
  scope: 通常 team
  内容: 外部系统入口或查找线索，不保存外部系统完整内容
```

不应保存到 Memory：

- 当前 Todo、短期计划、临时进度。
- API Key、token、密码、凭据。
- 可通过读取当前代码、README、Git 历史直接得到的信息。
- 只对当前版本成立的调试步骤。

## 加载路径一：索引进入 system prompt

位置：

- `managers/system_prompt_builder.py`
- `managers/memory_manager.py`

调用链：

```text
build_main_system_prompt(...)
-> _memory_index_prompt()
-> MEMORY_MANAGER.load_index_prompt()
```

`load_index_prompt()` 会读取：

```text
.memory/private/MEMORY.md
.memory/team/MEMORY.md
```

然后把索引和 `MEMORY_SYSTEM_RULES` 拼进 system prompt。

注意：这里注入的是索引，不是所有记忆正文。这样能让模型知道“有哪些记忆”，但不会把大量历史内容全部塞进 system prompt。

子 Agent 的 system prompt 也会调用 `build_subagent_system_prompt(...)`，所以它同样能看到 Memory 索引。

## 加载路径二：LLM side-query 按需注入正文

位置：

- `hooks/builtin_hooks.py`
- `managers/memory_manager.py`
- `prompts/memory_prompts.py`

Hook 注册顺序：

```text
BeforeModelCall -> memory_before_model_call_hook
BeforeModelCall -> todo_reminder_hook
BeforeModelCall -> compact_before_model_call_hook
BeforeModelCall -> log_before_model_call
```

执行流程：

```text
1. agent_loop 触发 BeforeModelCall。
2. memory_before_model_call_hook 调用 MEMORY_MANAGER.inject_relevant_memories(...)。
3. inject_relevant_memories 先移除上一轮注入的 [Memory context]。
4. select_relevant_memories 优先执行 LLM side-query。
5. side-query 接收最近对话和候选记忆目录。
6. 模型返回最多 MAX_RELEVANT_MEMORIES 个 filename。
7. MemoryManager 验证 filename 必须来自候选目录。
8. 读取选中的 .md 文件正文。
9. 作为一条 role=user 的 [Memory context] 临时追加到 messages。
10. 后续 todo_reminder_hook 和 compact_before_model_call_hook 继续处理。
```

side-query 使用的 prompt 是 `MEMORY_RELEVANCE_PROMPT`，要求模型只返回 JSON 字符串数组：

```json
["summary-autumn-article.md", "user-is-article-reader.md"]
```

如果 LLM 正常返回 `[]`，本轮不注入任何记忆正文。

如果 side-query 调用失败、JSON 解析失败或模型服务报错，会回退到关键词匹配：

```text
_select_relevant_memories_by_keyword(...)
```

这个兜底只用于保证系统可用，不是主路径。

## Memory 工具

Memory 工具定义在 `tools_configs/base_configs.py`，handler 在 `handlers/base_handlers.py`。

### save_memory

用于保存长期记忆。

参数：

```json
{
  "name": "user-is-article-reader",
  "scope": "private",
  "type": "user",
  "description": "用户是一位文章阅读人员，喜欢阅读文章并总结内容",
  "content": "Markdown 正文"
}
```

执行结果：

```text
已保存 private/user memory: user-is-article-reader
```

保存后会：

1. 生成安全 slug 文件名。
2. 写入 `.memory/private/*.md` 或 `.memory/team/*.md`。
3. 重建对应作用域的 `MEMORY.md`。
4. 标记 `saved_this_turn=True`，避免 Stop 阶段重复自动提取。

### forget_memory

用于删除错误、过时或用户明确要求忘记的记忆。

参数：

```json
{
  "name_or_path": "user-is-article-reader",
  "scope": "private",
  "reason": "用户要求删除"
}
```

执行结果：

```text
已删除 private memory: user-is-article-reader.md
```

删除后会重建对应作用域的 `MEMORY.md`。

## 自动提取

位置：

- `hooks/builtin_hooks.py`
- `managers/memory_manager.py`

触发点：

```text
Stop -> memory_stop_extract_hook
```

执行条件：

- `AUTO_EXTRACT_MEMORY: true`
- 当前不是禁止写入的子 Agent。
- 本轮没有主动调用 `save_memory` 或 `forget_memory`。
- Hook metadata 中有 `client` 和 `model_id`。

执行流程：

```text
1. 读取最近 EXTRACT_RECENT_MESSAGES 条消息。
2. 拼接已有记忆索引。
3. 使用 MEMORY_EXTRACT_PROMPT 调用模型。
4. 模型返回 JSON 数组。
5. 对每条结果调用 save_memory(...)。
6. 写入后重建索引。
7. 尝试触发 Dream。
```

自动提取只保存未来仍有价值的信息，不应该保存临时 Todo、短期计划、敏感信息或可从代码直接读取的信息。

## Dream

当前 `dream(...)` 是第一版实现：只做低频门控和索引重建。

触发条件：

- `DREAM_ENABLED: true`
- 记忆文件数量达到 `DREAM_MIN_FILES`
- `.memory/.dream-lock` 不存在或已过期

执行结果：

- 写入 `.memory/.dream-lock`。
- 重建 private/team 两个索引。
- 更新运行时 `last_dream_at`。

注意：`MEMORY_DREAM_PROMPT` 已经存在，但当前代码还没有调用 LLM 做合并、去重、修剪。真正的 LLM Dream 是后续增强。

## 子 Agent 行为

子 Agent 当前默认支持只读 Memory：

- 能看到 Memory 索引，因为 `build_subagent_system_prompt(...)` 也会注入 `_memory_index_prompt()`。
- 能通过 `BeforeModelCall` 注入相关 Memory 正文，因为 `ALLOW_SUBAGENT_MEMORY_READ: true`。
- 默认不能写 Memory，因为 `ALLOW_SUBAGENT_MEMORY_WRITE: false`。
- `tools_configs/subagent_configs.py` 没有把 `MEMORY_TOOLS` 加给子 Agent，所以子 Agent 不能主动调用 `save_memory` / `forget_memory`。

这种设计是有意的：子 Agent 可以遵守用户偏好和项目背景，但不应该把子任务中的临时判断写进长期记忆。

## 和 s08 Context Compact 的关系

Memory 注入发生在 `BeforeModelCall`，并且顺序早于 compact：

```text
memory_before_model_call_hook
todo_reminder_hook
compact_before_model_call_hook
log_before_model_call
```

这意味着：

- Memory context 会参与 s08 的上下文预算。
- 如果 Memory 正文过长，后续 compact 可以继续处理 messages。
- Compact 摘要不会替代长期 Memory；Memory 是跨会话的文件存储。

## 如何测试

### 测试索引注入

运行：

```powershell
python .\agent_loop.py
```

观察 system prompt 构建时是否读取 `.memory/private/MEMORY.md` 和 `.memory/team/MEMORY.md`。

### 测试相关记忆召回

准备一条 private memory，例如：

```text
.memory/private/user-is-article-reader.md
```

运行任务中提到“文章”“总结”等关键词。

日志应出现：

```text
LLM Memory 召回完成: selected=[...]
注入 Memory context: items=...
```

### 测试主动保存

让模型执行：

```text
请记住：我是文章阅读人员，喜欢结构化总结文章。
```

预期：

- 模型调用 `save_memory`。
- `.memory/private/` 下生成新的 `.md`。
- `.memory/private/MEMORY.md` 被重建。

### 测试删除

让模型执行：

```text
请忘记 user-is-article-reader 这条记忆。
```

预期：

- 模型调用 `forget_memory`。
- 对应 `.md` 被删除。
- `MEMORY.md` 被重建。

## 验收标准

- 启动时能自动初始化 `.memory/private/MEMORY.md` 和 `.memory/team/MEMORY.md`。
- system prompt 中只注入 Memory 索引，不注入所有正文。
- `BeforeModelCall` 能通过 LLM side-query 选择最多 5 个相关文件名。
- 选中的 Memory 正文会以 `[Memory context]` 注入当前 messages。
- side-query 失败时能回退关键词匹配。
- `save_memory` 能保存 private/team 记忆并重建索引。
- `forget_memory` 能删除记忆并重建索引。
- Stop 阶段能自动提取长期有效信息。
- 子 Agent 默认只读 Memory，不写 Memory。
- s08 Context Compact 正常处理注入后的 Memory context。
