# s03：权限控制——执行前做权限判断

s01 → s02 → `s03` → s04 → s05 → ... → s20

> 工具调用不能只相信模型的判断。s03 在工具真正执行前增加权限检查，先判断能不能执行，再交给具体工具函数。

---

## 问题

s02 已经让 Agent 可以调用 `bash`、`read_file`、`write_file`、`edit_file`。这让 Agent 能完成更多任务，但也带来风险：

- 模型可能用 `bash` 执行删除命令，例如 `del summary.txt /f /q`。
- 模型可能尝试读取项目外文件。
- 模型可能写入或编辑不该修改的文件。

所以工具执行前需要一层统一权限控制。权限控制不写进每个工具函数里，而是放在 `handlers/dispatcher.py` 执行工具前统一处理。

---

## 当前文件

```text
configs/
└── permission_config.yml

state/
└── permission_state.py

utils/
├── config_handler.py
├── path_sandbox.py
└── permission_check.py

handlers/
└── dispatcher.py
```

各文件职责：

- `configs/permission_config.yml`：配置允许工具、拒绝工具、危险 bash 命令、审批模式。
- `state/permission_state.py`：定义 `PermissionResult`，统一表达权限结果。
- `utils/config_handler.py`：读取 YAML 配置，并提供默认权限配置。
- `utils/permission_check.py`：实现三道闸门权限控制。
- `handlers/dispatcher.py`：在工具执行前调用 `check_permission()`。
- `utils/path_sandbox.py`：保留 s02 的路径沙箱，作为文件工具的最后防线。

## 配置文件

当前 `configs/permission_config.yml`：

```yaml
tools:
  allow:
    - bash
    - read_file
    - write_file
    - edit_file
  deny: []

dangerous_commands:
  - "rm -rf"
  - "rm"
  - "del"
  - "erase"
  - "rd"
  - "rmdir"
  - "Remove-Item"
  - "Format-Volume"
  - "shutdown"
  - "reboot"

approval:
  mode: "ask"
  required_tools:
    - write_file
    - edit_file
```

说明：

- `tools.allow` 是模型可以调用的工具列表。
- `tools.deny` 是强制拒绝工具列表，优先级高于 `allow`。
- `dangerous_commands` 是 bash 的危险命令关键字。
- `approval.mode` 控制审批方式：
  - `ask`：询问用户。
  - `auto_allow`：测试时自动允许。
  - `auto_deny`：测试时自动拒绝。
- `approval.required_tools` 是默认需要审批的工具。

注意：当前配置里没有 `paths.read` 和 `paths.write`。读写路径策略由 `permission_check.py` 直接表达：

- `read_file`：当前项目内直接允许，项目外进入审批。
- `write_file` 和 `edit_file`：不管路径在哪里，都先询问用户。

## 三道闸门

权限入口是 `utils/permission_check.py` 中的 `check_permission(tool_name, args, config=None)`。

### 闸门 1：拒绝列表

第一步先看是否必须拒绝：

- 工具名是否在 `tools.deny`。
- bash 命令是否命中 `dangerous_commands`。

危险 bash 命令直接拒绝，不进入用户审批。这样可以避免模型通过删除命令破坏文件。

当前实现不是简单匹配固定字符串，而是对单个命令词做边界匹配。例如配置了 `del` 后，下面两种命令都会被拒绝：

```text
del summary.txt /f /q
cmd /c del /f /q summary.txt
```

如果只配置 `del /s`，上面的命令不会命中，这就是之前删除文件会被放行的原因。

### 闸门 2：规则匹配

第二步判断工具是否属于当前 Agent 允许使用的能力：

- 如果 `tools.allow` 非空，工具名必须出现在 `allow` 里。
- `read_file` 会先用 `safe_path()` 判断路径是否在当前项目内。
- 当前项目内的读取直接允许。

这一步解决的是“模型能不能调用这个工具”和“读文件是不是普通项目内读取”。

### 闸门 3：用户审批

第三步处理需要人工确认的操作：

- `read_file` 读取项目外路径时，需要询问用户。
- `write_file` 总是需要询问用户。
- `edit_file` 总是需要询问用户。
- `approval.required_tools` 中的其他工具也会进入审批。

审批函数是 `request_user_approval()`。如果用户输入 `y` 或 `yes`，权限通过；如果用户拒绝、直接回车、终端无法输入或中断，权限失败。

## Dispatcher 如何接入

工具调用统一经过 `handlers/dispatcher.py`。

流程是：

1. `_get_tool_name_and_args()` 解析 OpenAI 返回的 tool call。
2. 从 `BASE_HANDLERS` 查找工具处理函数。
3. 调用 `check_permission(name, args)`。
4. 如果 `allowed=False`，返回 `Permission denied: ...`，不执行工具。
5. 如果权限通过，才调用真实 handler。

关键逻辑：

```python
permission = check_permission(name, args)
if not permission.allowed:
    logger.warning("权限拒绝: %s, 原因: %s", name, permission.reason)
    return f"Permission denied: {permission.reason}"

result = handler(args)
```

这样权限判断集中在 dispatcher，不需要散落到 `bash_tools.py`、`file_tools.py` 里。

## 路径沙箱和权限审批的关系

当前项目保留了 s02 的 `safe_path()`：

```python
WORKDIR = Path.cwd().resolve()

def safe_path(path: str) -> Path:
    resolved = (WORKDIR / path).resolve()
    resolved.relative_to(WORKDIR)
    return resolved
```

这意味着：

- 权限层可以决定“是否允许尝试执行”。
- 工具层仍会决定“实际路径是否能访问”。
- 即使项目外读取在权限层审批通过，`read_file()` 里的 `safe_path()` 仍会阻止访问项目外文件。

这是一种双层保护：权限层负责决策，工具层负责最后的执行安全。

## 可以验证的行为

### 1. 危险 bash 被拒绝

```python
check_permission("bash", {"command": "cmd /c del /f /q summary.txt"})
```

结果：

```text
PermissionResult(allowed=False, reason='dangerous command keyword: del', needs_approval=False)
```

### 2. 普通 bash 允许

```python
check_permission("bash", {"command": "Get-ChildItem"})
```

结果：

```text
PermissionResult(allowed=True, reason='permission allowed', needs_approval=False)
```

### 3. 项目内读文件直接允许

```python
check_permission("read_file", {"path": "data/秋天.txt"})
```

结果：

```text
PermissionResult(allowed=True, reason='read path inside project', needs_approval=False)
```

### 4. 写文件需要审批

```python
check_permission("write_file", {"path": "data/result.txt", "content": "hello"})
```

在 `approval.mode: ask` 下会询问用户。

## 相对 s02 的变化

| 位置 | s02 | s03 |
|------|-----|-----|
| `dispatcher.py` | 找到 handler 后直接执行 | 执行前先调用 `check_permission()` |
| `permission_config.yml` | 无 | 统一配置工具 allow/deny、危险命令、审批模式 |
| `permission_check.py` | 无 | 三道闸门权限控制 |
| `permission_state.py` | 无 | 用 `PermissionResult` 表达权限结果 |
| 文件工具 | 只有 `safe_path()` | `safe_path()` 仍保留，权限层在执行前先判断 |

## 当前完成标准

- 删除类 bash 命令会被拒绝。
- allow/deny 工具配置生效。
- 当前项目内读文件直接允许。
- 项目外读文件进入审批。
- 写文件和编辑文件统一进入审批。
- 用户拒绝时，dispatcher 不会调用真实工具函数。
- 权限逻辑集中在 `utils/permission_check.py` 和 `handlers/dispatcher.py`。
