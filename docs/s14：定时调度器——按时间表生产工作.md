# s14：定时调度器——按时间表生产工作

> **按时间表生产工作，调度与执行解耦。**
>
> Cron Scheduler 是 Harness 层能力：独立线程判断时间，到点后把工作事件放进队列；队列处理器只在 Agent 空闲时交付；Agent Loop 只消费已触发的工作，不负责检查时间。

---

## 问题

s13 解决的是“慢操作不要阻塞主循环”。模型可以把 `pytest`、构建、安装依赖这类慢命令放到后台执行。

但 s13 的触发仍然来自用户输入。用户说一句，Agent 动一下。

下面这些工作不应该每次都靠人手动触发：

- 每天 9 点运行测试并总结失败项。
- 每 30 分钟检查一次构建状态。
- 每 5 分钟读取当前目录并生成简短状态报告。
- 创建一个一次性提醒，到点后让 Agent 检查某个任务。

s14 的目标是让 Agent 进程运行期间，调度器可以按时间表自动生产工作。

## 四层模型

Cron Scheduler 分四层：

```text
Scheduler
  独立 daemon 线程。
  每隔 CHECK_INTERVAL_SECONDS 检查 cron 表达式是否到点。
  到点后只生成 FiredCronEvent，并写入内存队列。

Queue
  进程内 deque。
  Scheduler 写入，Queue Processor 消费。

Queue Processor
  独立 daemon 线程。
  每隔 QUEUE_PROCESSOR_INTERVAL_SECONDS 检查队列。
  队列非空且 Agent 空闲时，拉起一轮 scheduled agent turn。

Agent Consumer
  run_scheduled_agent_turn 消费 FiredCronEvent。
  格式化为 <scheduled_task> user message。
  调用现有 agent_loop 执行。
```

职责边界：

| 层 | 负责 | 不负责 |
|---|---|---|
| Scheduler | 判断时间、去重、入队 | 调模型、执行工具、修改 messages |
| Queue | 暂存已触发事件 | 判断 cron 是否到点 |
| Queue Processor | 空闲时交付事件 | 解析 cron、决定任务是否触发 |
| Agent Loop | 执行已交付 prompt | 主动扫描定时任务 |

这就是“调度与执行解耦”：Cron 只负责什么时候生产工作，Agent 负责怎么完成工作。

## 数据结构

`state/cron_state.py` 定义两个 dataclass。

### CronJobRecord

```python
@dataclass
class CronJobRecord:
    id: str
    cron: str
    prompt: str
    recurring: bool = True
    durable: bool = True
    enabled: bool = True
    created_at: str = ""
    updated_at: str = ""
    last_fired_at: str = ""
    last_fire_marker: str = ""
    fire_count: int = 0
    metadata: Dict = field(default_factory=dict)
```

`CronJobRecord` 是任务定义。它描述什么时候触发、触发后给 Agent 什么 prompt、是否重复、是否持久化。

### FiredCronEvent

```python
@dataclass
class FiredCronEvent:
    event_id: str
    job_id: str
    cron: str
    prompt: str
    fired_at: str
    durable: bool
    recurring: bool
```

`FiredCronEvent` 是已经触发、等待交付的工作事件。调度器入队的是 event，而不是直接执行 job。

## Cron 表达式

当前实现支持标准五段式：

```text
分钟 小时 日 月 星期
```

示例：

```text
* * * * *        每分钟
*/5 * * * *      每 5 分钟
0 9 * * *        每天 9:00
0 9 * * 1-5      工作日 9:00
```

支持语法：

- `*`
- `*/N`
- `N`
- `N-M`
- `N-M/S`
- `N,M,...`

不支持：

- 秒级 cron
- `L`
- `W`
- `?`

时间使用本地时区 `datetime.now().astimezone()`。

day-of-week 使用 cron 语义：Sunday = 0 或 7，Monday = 1。day-of-month 和 day-of-week 同时受限时使用 OR 语义。

创建任务前会调用 `validate_cron(expr)`。加载 durable 任务时也会校验，坏任务会被跳过并写日志，不会拖垮启动。

## CronManager

核心实现位于 `managers/cron_manager.py`，模块底部创建全局单例：

```python
CRON_MANAGER = CronManager()
```

主要方法：

| 方法 | 作用 |
|---|---|
| `ensure_store()` | 初始化 `.runtime-tasks/`、任务文件和事件日志 |
| `start(client, model_id, hook_manager, workdir, runner)` | 幂等启动 scheduler 和 queue processor |
| `create(cron, prompt, recurring=True, durable=True)` | 创建定时任务 |
| `cancel(cron_id)` | 取消定时任务 |
| `list_all(include_session=True)` | 列出 durable 和 session-only 任务 |
| `validate_cron(expr)` | 校验 cron 表达式 |
| `cron_matches(expr, dt)` | 判断某个时间是否匹配表达式 |
| `enqueue_due_jobs(now)` | 到点任务转为 `FiredCronEvent` 入队 |
| `consume_triggered_events(limit)` | 消费队列事件 |
| `format_events_as_user_message(events)` | 生成 `<scheduled_task>` 消息 |

内部关键状态：

```python
self.lock = threading.RLock()
self.agent_lock = threading.Lock()
self.durable_jobs: Dict[str, CronJobRecord] = {}
self.session_jobs: Dict[str, CronJobRecord] = {}
self.queue: Deque[FiredCronEvent] = deque()
self.runner: Optional[AgentRunner] = None
```

`agent_lock` 很重要。它保证手动用户请求和 cron 自动请求不会同时驱动同一个 Agent Loop。

## 启动流程

`agent_loop.py` 中：

```python
hook_manager = get_default_hook_manager()
hook_manager.run_hooks("SessionStart", HookContext(event="SessionStart"))
client, model_id = build_model()
CRON_MANAGER.start(
    client=client,
    model_id=model_id,
    hook_manager=hook_manager,
    workdir=str(WORKDIR),
    runner=run_scheduled_agent_turn,
)
```

`SessionStart` 阶段会执行：

```python
def cron_session_start_hook(context: HookContext):
    CRON_MANAGER.ensure_store()
    CRON_MANAGER.load_durable_jobs()
```

`CRON_MANAGER.start(...)` 会：

1. 保存 `client`、`model_id`、`hook_manager` 和 `runner`。
2. 再次确保存储存在并加载 durable 任务。
3. 幂等启动 scheduler 线程。
4. 幂等启动 queue processor 线程。

## 手动执行与定时执行的锁

`agent_loop.py` 的手动入口：

```python
with CRON_MANAGER.agent_execution():
    answer = agent_loop(state, client, model_id, hook_manager=hook_manager)
```

queue processor 自动交付时也会尝试获取同一把 `agent_lock`：

```python
if not self.agent_lock.acquire(blocking=False):
    continue
```

效果：

- 用户手动请求正在执行时，cron 到点不会并发启动第二轮 Agent。
- cron 自动执行正在运行时，手动入口也不会和它共享同一套状态同时跑。
- 避免 messages、日志、runtime 文件、工具结果交错。

## 到点后如何交付

`run_scheduled_agent_turn(...)` 是 CronManager 调用 Agent Loop 的回调：

```python
def run_scheduled_agent_turn(client, model_id: str, hook_manager=None) -> str:
    events = CRON_MANAGER.consume_triggered_events(
        limit=cron_config.get("MAX_TRIGGERED_EVENTS_PER_TURN", 5)
    )
    if not events:
        return ""

    message = CRON_MANAGER.format_events_as_user_message(events)
    state = LoopState(messages=[{"role": "user", "content": message}])
    return agent_loop(state, client, model_id, hook_manager=hook_manager)
```

注入给 Agent 的消息格式：

```xml
<scheduled_task>
  <event_id>cron_evt_000001</event_id>
  <cron_id>cron_000001</cron_id>
  <cron>*/5 * * * *</cron>
  <fired_at>2026-07-08T20:50:00+08:00</fired_at>
  <prompt>检查当前目录文件列表，并总结是否有新增文件。</prompt>
</scheduled_task>
```

Agent 收到这条消息后，像处理普通用户请求一样决定调用哪些工具。

## 示例流程

以“每 5 分钟检查当前目录”为例。

### 1. 用户提出需求

```text
请每 5 分钟检查当前目录文件列表，并总结是否有新增文件。
```

### 2. 模型调用 schedule_cron

```json
{
  "cron": "*/5 * * * *",
  "prompt": "检查当前目录文件列表，并总结是否有新增文件。",
  "recurring": true,
  "durable": true
}
```

### 3. CronManager 创建任务

`CRON_MANAGER.create(...)` 会：

- 校验 cron 表达式。
- 检查任务数量是否超过 `MAX_CRON_JOBS`。
- 分配 `cron_000001`。
- 写入 `.runtime-tasks/scheduled_tasks.json`。
- 向 `.runtime-tasks/scheduled_events.jsonl` 追加 `create` 事件。

### 4. 到点触发

scheduler 线程每秒调用：

```python
enqueue_due_jobs(datetime.now().astimezone())
```

当 `cron_matches("*/5 * * * *", now)` 为 true，且 `last_fire_marker` 不是当前分钟时：

- 生成 `FiredCronEvent`
- append 到 `self.queue`
- 更新 `last_fire_marker`
- 更新 `last_fired_at`
- `fire_count += 1`
- 追加 `fire` 事件日志

### 5. 空闲时交付

queue processor 线程发现 `self.queue` 非空：

```text
Agent 忙 -> 获取 agent_lock 失败 -> 等下一轮检查
Agent 空闲 -> 获取 agent_lock 成功 -> 调用 runner
```

### 6. Agent 执行

`runner=run_scheduled_agent_turn` 消费事件，生成 `<scheduled_task>`，再调用 `agent_loop(...)`。

Agent 看到 prompt 后可以调用：

```json
{"command": "Get-ChildItem"}
```

如果定时 prompt 要求跑完整测试，Agent 也可以继续使用 s13 后台任务：

```json
{
  "command": "pytest",
  "run_in_background": true
}
```

Cron Scheduler 不直接决定怎么执行命令；它只负责把工作按时间送进 Agent。

## Durable vs Session-only

- Durable：写入 `.runtime-tasks/scheduled_tasks.json`，Agent 重启后会加载任务定义。
- Session-only：只保存在当前进程的 `self.session_jobs`，进程退出后丢失。

重要边界：

- 进程关闭后不会继续触发任务。
- Durable 只代表任务定义跨重启保留，不是系统级定时器。
- 如果需要进程关闭后也能触发，请使用系统 crontab、systemd timer 或 Windows Task Scheduler。

---

## 和 s13 Background Tasks 的关系

Cron Scheduler 负责“什么时候产生工作”。

Background Tasks 负责“慢命令怎么异步执行”。

两者不直接耦合：

```text
Cron 到点
→ 注入 <scheduled_task>
→ Agent 理解 prompt
→ Agent 决定是否调用 bash(run_in_background=true)
→ BackgroundManager 执行慢命令并注入 <task_notification>
```

不要让 CronManager 直接调用 `BACKGROUND_MANAGER.run_command(...)`。否则调度层会越过 Agent 决策，破坏“调度与执行解耦”。

---

## 验收

- `schedule_cron` 能创建 durable 和 session-only 任务。
- `list_crons` 能列出任务 ID、cron、recurring/durable、last fired、fire count。
- `cancel_cron` 能取消任务。
- cron 表达式非法时拒绝创建。
- durable 任务写入 `.runtime-tasks/scheduled_tasks.json`。
- session-only 任务不写入 durable 文件。
- 到点后 scheduler 只入队，不直接调用模型。
- queue processor 在 Agent 空闲时自动交付。
- Agent 收到 `<scheduled_task>` 后像普通用户请求一样执行。
- 同一分钟内同一任务不会重复触发。
- 第二天同一分钟可以再次触发。
- 一次性任务触发后自动删除。
- 单个坏 job 不会杀掉 scheduler 线程。
- 手动 Agent 执行中，cron 不会并发启动另一轮 Agent。
- Agent 进程关闭后不触发任务，文档和 prompt 都要明确这一点。

## 设计结论

s14 的关键不是“多一个定时工具”，而是把三个角色拆开：

```text
Scheduler 生产事件
Queue 保存事件
Queue Processor 空闲时交付
Agent Loop 消费并执行
```

这样 Cron Scheduler 可以复用当前项目已有的工具系统、权限系统、Hook、错误恢复、上下文压缩、Memory 和 Background Tasks，同时保持调度层足够简单。
