# s13：后台任务——慢操作放后台

s01 → ... → s11 → s12 → `s13` → s14 → ... → s20

> **慢操作丢后台，Agent 继续处理。**
>
> Background Tasks 是 Harness 层能力：后台线程跑命令，主循环不等待，完成后把通知注入到后续模型上下文。

---

## 问题

普通 `bash` 工具是同步执行的。模型调用：

```json
{"command": "pytest"}
```

Agent 就会一直等到命令结束，才能进入下一轮。

对于 `pwd`、`git status`、`dir` 这种短命令，同步等待没问题。但下面这些命令可能持续很久：

- `npm install`
- `pip install`
- `pytest`
- `npm run build`
- `docker build`

如果 Agent 一直等慢命令结束，就不能同时读取文件、整理计划、查看任务看板或继续处理其他上下文。

s13 的目标是让模型显式把慢命令放后台：

```json
{
  "command": "pytest",
  "run_in_background": true
}
```

工具立即返回后台任务 ID，例如 `bg_000001`。命令在线程里继续跑，完成后由 Hook 注入 `<task_notification>`。

## 同步和后台的区别

| 能力 | 同步 bash | 后台 bash |
|---|---|---|
| 参数 | `command` | `command` + `run_in_background=true` |
| 主循环 | 等命令结束 | 立即继续 |
| tool result | 返回完整命令结果 | 返回后台任务 ID 和输出路径 |
| 完成结果 | 当前轮可见 | 后续轮次以通知注入 |
| 输出 | 直接进入上下文 | 写入 `.runtime-tasks/outputs/` |

后台任务不是“命令已经完成”。它只表示命令已经启动。

## 数据模型

`state/background_state.py` 定义后台任务状态：

```python
BACKGROUND_STATUSES = {"running", "completed", "failed", "cancelled"}

@dataclass
class BackgroundTaskRecord:
    id: str
    command: str
    status: str = "running"
    exit_code: Optional[int] = None
    output_path: str = ""
    owner: str = "main"
    started_at: str = ""
    finished_at: str = ""
    summary: str = ""
    metadata: Dict = field(default_factory=dict)
    notified: bool = False
```

字段说明：

| 字段 | 说明 |
|---|---|
| `id` | 后台任务 ID，例如 `bg_000001` |
| `command` | 实际运行的命令 |
| `status` | `running` / `completed` / `failed` / `cancelled` |
| `exit_code` | 命令退出码，运行中为 `null` |
| `output_path` | 输出文件路径 |
| `owner` | 发起者，默认 `main` |
| `started_at` / `finished_at` | ISO 时间戳 |
| `summary` | 输出摘要，用于通知 |
| `metadata` | 扩展字段 |
| `notified` | 完成通知是否已经注入过 |

## 持久化目录

默认配置在 `configs/background_config.yml`：

```yaml
ENABLE_BACKGROUND_TASKS: true
RUNTIME_TASK_DIR: ".runtime-tasks"
OUTPUT_DIR: ".runtime-tasks/outputs"
TASK_ID_PREFIX: "bg_"
MAX_NOTIFICATION_OUTPUT_CHARS: 1200
MAX_BACKGROUND_TASKS_IN_LIST: 50
DEFAULT_COMMAND_TIMEOUT_SECONDS: 0
SLOW_COMMAND_KEYWORDS:
  - "npm install"
  - "pip install"
  - "pytest"
  - "npm test"
  - "npm run build"
  - "docker build"
  - "cargo build"
  - "make"
```

运行时目录：

```text
.runtime-tasks/
├── index.json
├── events.jsonl
└── outputs/
    └── bg_000001.txt
```

`index.json` 保存任务索引：

```json
{
  "version": 1,
  "next_id": 2,
  "tasks": {
    "bg_000001": {
      "id": "bg_000001",
      "command": "Start-Sleep -Seconds 5; Write-Output background done",
      "status": "completed",
      "exit_code": 0,
      "output_path": ".runtime-tasks\\outputs\\bg_000001.txt",
      "owner": "main",
      "started_at": "2026-07-07T22:50:21+08:00",
      "finished_at": "2026-07-07T22:50:26+08:00",
      "summary": "shell: powershell command: ... stdout: background done",
      "metadata": {},
      "notified": true
    }
  }
}
```

`events.jsonl` 追加事件：

```jsonl
{"time":"2026-07-07T22:50:21+08:00","event":"start","bg_id":"bg_000001","command":"Start-Sleep -Seconds 5; Write-Output background done","owner":"main"}
{"time":"2026-07-07T22:50:26+08:00","event":"complete","bg_id":"bg_000001","exit_code":0,"summary":"shell: powershell ..."}
{"time":"2026-07-07T22:50:26+08:00","event":"notify","bg_id":"bg_000001","status":"completed"}
```

输出文件保存完整命令结果：

```text
shell: powershell

command: Start-Sleep -Seconds 5; Write-Output background done

exit_code: 0

stdout:
background done
```

## BackgroundManager

核心实现位于 `managers/background_manager.py`。

主要方法：

| 方法 | 作用 |
|---|---|
| `ensure_store()` | 创建 `.runtime-tasks/`、`outputs/`、`index.json`、`events.jsonl` |
| `run_command(...)` | 创建 `bg_id`，启动 daemon 线程，立即返回占位结果 |
| `_worker(...)` | 在线程里运行命令，写输出文件，更新状态 |
| `list_all(...)` | 列出后台任务 |
| `get(bg_id, tail_chars=2000)` | 返回单个任务状态和输出尾部 |
| `drain_notifications()` | 取出已完成但未通知的任务，生成 `<task_notification>` |
| `append_event(...)` | 追加事件日志 |

模块底部创建全局单例：

```python
BACKGROUND_MANAGER = BackgroundManager()
```

`base_handlers.py` 和 Hook 都直接使用这个单例。

## 后台执行流程

模型调用：

```json
{
  "command": "Start-Sleep -Seconds 5; Write-Output background done",
  "run_in_background": true,
  "background_owner": "main"
}
```

`handlers/base_handlers.py` 中的 `handle_bash` 分支：

```python
def handle_bash(args: dict) -> str:
    if args.get("run_in_background"):
        return BACKGROUND_MANAGER.run_command(
            command=args["command"],
            owner=args.get("background_owner", "main"),
        )
    return run_bash(args["command"])
```

后台启动后立即返回：

```text
后台任务已启动: bg_000001
command: Start-Sleep -Seconds 5; Write-Output background done
status: running
output_path: .runtime-tasks\outputs\bg_000001.txt
后续完成后会以 <task_notification> 注入。
```

这条返回是原始 `bash` tool call 的 tool result。它只说明后台任务已启动，不说明命令完成。

## 命令执行

后台线程使用：

```python
threading.Thread(
    target=self._worker,
    args=(bg_id, command, cwd, timeout),
    daemon=True,
    name=f"background-{bg_id}",
)
```

命令 shell 选择复用 `tools.bash_tools._shell_command()`：

- Windows：`powershell -NoProfile -ExecutionPolicy Bypass -Command ...`
- Linux/macOS：`bash -lc ...`

后台线程中使用 `subprocess.run(...)` 执行命令：

```python
result = subprocess.run(
    command_args,
    shell=False,
    cwd=effective_cwd,
    capture_output=True,
    text=True,
    encoding="utf-8",
    errors="replace",
    timeout=timeout_value,
)
```

执行完成后：

- `returncode == 0` → `status = "completed"`
- `returncode != 0` → `status = "failed"`
- 超时或异常 → `status = "failed"`

无论成功失败，输出都会写入 `.runtime-tasks/outputs/<bg_id>.txt`。

## 通知注入

后台完成不是 tool result。原始 tool call 已经返回了“后台任务已启动”。

完成事件通过 Hook 注入。`hooks/builtin_hooks.py` 中：

```python
def background_before_model_call_hook(context: HookContext):
    if context.messages is None:
        return None
    if int(context.metadata.get("subagent_depth", 0)) > 0:
        return None

    notifications = BACKGROUND_MANAGER.drain_notifications()
    if not notifications:
        return None

    context.messages.append({
        "role": "user",
        "content": "\n\n".join(notifications),
    })
    return None
```

`hooks/hook_manager.py` 在 `BeforeModelCall` 注册它：

```python
self.register("BeforeModelCall", builtin_hooks.background_before_model_call_hook)
```

注入格式：

```xml
<task_notification>
  <task_id>bg_000001</task_id>
  <status>completed</status>
  <command>Start-Sleep -Seconds 5; Write-Output background done</command>
  <exit_code>0</exit_code>
  <output_path>.runtime-tasks\outputs\bg_000001.txt</output_path>
  <summary>shell: powershell command: ... stdout: background done</summary>
</task_notification>
```

`drain_notifications()` 会把 `notified` 改成 `true`，同一个后台任务只通知一次。

## 工具

### bash

`tools_configs/base_configs.py` 的 `bash` schema 增加两个可选参数：

```json
{
  "command": "命令",
  "run_in_background": true,
  "background_owner": "main"
}
```

默认不后台执行：

```json
{"command": "git status"}
```

只有模型显式传 `run_in_background=true` 时才进入后台。

当前代码没有自动关键词后台化。`SLOW_COMMAND_KEYWORDS` 只是配置预留和 prompt 提示依据，不会自动改变执行策略。

### background_list

列出后台任务：

```json
{
  "include_completed": true
}
```

返回示例：

```text
后台任务：1/1
- bg_000001 [completed] exit=0 owner=main output=.runtime-tasks\outputs\bg_000001.txt :: Start-Sleep -Seconds 5; Write-Output background done
```

### background_get

读取指定后台任务详情：

```json
{
  "bg_id": "bg_000001",
  "tail_chars": 2000
}
```

返回 JSON，包含状态、输出路径和 `output_tail`。

## Prompt 接入

`prompts/background_prompts.py`：

```python
BACKGROUND_SYSTEM_RULES = """后台任务规则：
- 预计耗时较长的命令可以设置 bash.run_in_background=true，例如安装依赖、运行完整测试、构建、部署、docker build。
- 后台任务启动后不要声称已经完成，只能说已启动并记录 bg_id。
- 后台任务完成通知会以 <task_notification> 注入；看到通知后再根据 exit_code 和 summary 判断成功或失败。
- 需要主动查看后台任务时调用 background_list 或 background_get。
- 不要把快速命令放后台，例如 pwd、ls、git status、读取小文件。
"""
```

`prompts/prompt_sections.py` 增加 `background_rules`。

`managers/system_prompt_builder.py` 中：

```python
def _background_rules_prompt(agent_type: str) -> str:
    if not background_config.get("ENABLE_BACKGROUND_TASKS", True):
        return ""
    if agent_type == "subagent":
        return ""
    return BACKGROUND_SYSTEM_RULES
```

所以：

- 主 Agent 默认看到后台任务规则。
- 子 Agent 默认不注入后台任务规则。

## 和 Task System 的关系

s12 Task System 管任务图和进度。

s13 Background Tasks 管慢命令怎么异步执行。

两者不自动耦合：

- 后台任务完成后，不会自动把 `.tasks/tasks/*.json` 里的任务改成 `completed`。
- 模型看到 `<task_notification>` 后，可以自己判断是否调用 `task_complete`。
- 后续可以把 `bg_id` 放入 Task System 的 `metadata`，但当前代码没有自动绑定。

---

## 重要边界

当前实现使用进程内 daemon 线程。

这意味着：

- Agent 进程还在时，后台命令可以继续跑。
- Agent 主循环可以继续处理其他工具。
- Agent 进程退出后，daemon 线程会被终止。
- `.runtime-tasks/index.json` 和输出文件会保留，但不能让已经被杀掉的线程继续执行。

如果后续需要“Agent 退出后后台命令仍然继续跑”，需要改成独立 subprocess + PID 管理，不是当前 s13 的最小实现。

---

## 错误处理

后台命令失败不会杀掉 Agent 主循环。

失败场景：

- 命令退出码非 0。
- 命令超时。
- 后台线程执行异常。

这些都会记录为 `status = "failed"`，并且仍然注入 `<task_notification>`。

例如：

```xml
<task_notification>
  <task_id>bg_000002</task_id>
  <status>failed</status>
  <command>bad-command</command>
  <exit_code>1</exit_code>
  <output_path>.runtime-tasks\outputs\bg_000002.txt</output_path>
  <summary>shell: powershell ... stderr: ...</summary>
</task_notification>
```

`background_get` 如果读取不存在的 `bg_id`，会抛出清晰错误，由 `dispatcher.py` 捕获并作为 tool result 返回给模型自纠。

可以在 `agent_loop.py` 中临时使用：

```python
question = """
请测试 s13 Background Tasks 功能，不要修改项目源代码，不要调用 write_file/edit_file。

请按顺序完成：
1. 调用 bash 启动一个后台命令，必须设置 run_in_background=true：
   Windows 环境使用命令：Start-Sleep -Seconds 5; Write-Output "background done"
2. 记录返回的 bg_id，并明确说明这个后台任务只是已启动，还没有完成。
3. 在后台任务运行期间，继续调用 read_file 读取 configs/background_config.yml，证明 Agent 没有阻塞等待后台命令完成。
4. 调用 background_list 查看后台任务状态。
5. 如果还没有收到 <task_notification>，继续进行下一轮等待通知或再次调用 background_list。
6. 收到 <task_notification> 后，调用 background_get 查看对应 bg_id 的详情和输出尾部。
7. 最后用中文总结：
   - 后台任务 ID 是什么
   - 启动后是否立即返回
   - Agent 是否在后台任务运行期间继续执行了 read_file
   - 是否收到了 <task_notification>
   - 输出文件路径是什么
   - background_get 是否能看到 "background done"
"""
```

观察重点：

- `bash.run_in_background=true` 是否立即返回 `bg_id`。
- 后台运行期间是否还能执行 `read_file`。
- `.runtime-tasks/outputs/<bg_id>.txt` 是否生成。
- 后续模型调用前是否注入 `<task_notification>`。
- `background_get` 是否能读取输出尾部。

## 验收

- `bash` 默认同步执行，兼容旧行为。
- `bash(run_in_background=true)` 能立即返回后台任务 ID。
- 后台任务输出写入 `.runtime-tasks/outputs/`。
- 后台任务完成后状态变为 `completed` 或 `failed`。
- `BeforeModelCall` 能注入 `<task_notification>`。
- 同一个后台任务只通知一次。
- `background_list` 能列出后台任务。
- `background_get` 能读取单个任务状态和输出尾部。
- 子 Agent 默认不注入后台任务规则。
- Agent 进程退出后后台线程不保证继续运行，这一点在文档中明确说明。
