# s12：任务系统——目标太大，拆成小任务

s01 → ... → s10 → s11 → `s12` → s13 → s14 → ... → s20

> **大目标拆成小任务，任务之间按依赖排序，每个任务落盘保存。**
>
> Task System 是 Harness 层能力：持久化目标、可恢复进度，并为后续多 Agent 协作提供任务看板。

---

## 问题

s05 的 `todo` 只能管理当前会话里的短期计划。它适合提醒模型“现在做到哪一步”，但不适合保存一个跨会话、可恢复、可被多个 Agent 协作推进的大目标。

例如用户要求：

- 先设计数据模型。
- 再实现创建、列表、读取工具。
- 再实现认领、完成、解锁逻辑。
- 最后写验收说明。

这些工作有依赖关系，不能只靠一张内存里的 todo。程序重启后，Agent 也应该能通过磁盘文件恢复任务状态。

所以 s12 引入 Task System：把大目标拆成多个持久任务，每个任务是一个 JSON 文件，任务之间通过 `blocked_by` / `blocks` 形成依赖图。

## Todo 和 Task System 的边界

| 能力 | `todo` | Task System |
|---|---|---|
| 定位 | 当前会话短期计划 | 跨会话持久任务图 |
| 存储 | 内存里的 `PlanningState` | `.tasks/` 文件 |
| 依赖 | 无 | `blocked_by` / `blocks` |
| owner | 无 | 有 |
| 重启恢复 | 不能 | 可以 |
| 多 Agent 协作 | 不适合 | 适合作共享看板 |

`todo` 仍然有用。它负责当前认领任务里的短期执行步骤。

Task System 负责项目级任务图：有哪些任务、谁在做、依赖是否完成、哪些任务被解锁。

## 数据模型

`state/task_state.py` 定义 `TaskRecord`：

```python
@dataclass
class TaskRecord:
    id: str
    subject: str
    description: str = ""
    status: str = "pending"
    owner: Optional[str] = None
    active_form: str = ""
    blocked_by: List[str] = field(default_factory=list)
    blocks: List[str] = field(default_factory=list)
    metadata: Dict = field(default_factory=dict)
    created_at: str = ""
    updated_at: str = ""
```

状态集合：

```python
TASK_STATUSES = {"pending", "in_progress", "completed", "cancelled"}
```

字段含义：

| 字段 | 说明 |
|---|---|
| `id` | 稳定任务 ID，例如 `task_000001` |
| `subject` | 一行任务标题 |
| `description` | 完整任务说明，用于跨会话恢复 |
| `status` | `pending` / `in_progress` / `completed` / `cancelled` |
| `owner` | 当前认领者，例如 `main`、`test_user`、`agent-a` |
| `active_form` | 进行时短语，供展示使用 |
| `blocked_by` | 上游依赖任务 ID 列表 |
| `blocks` | 下游任务 ID 列表，由 `TaskManager` 自动重建 |
| `metadata` | 扩展信息，例如完成摘要、取消原因，后续可放后台任务或 worktree 信息 |
| `created_at` / `updated_at` | ISO 时间戳 |

## 持久化目录

默认配置在 `configs/task_config.yml`：

```yaml
ENABLE_TASK_SYSTEM: true
TASK_DIR: ".tasks"
TASK_ID_PREFIX: "task_"
ENABLE_TASK_LOCKS: true
LOCK_TIMEOUT_SECONDS: 5
MAX_TASKS_IN_LIST_OUTPUT: 50
ALLOW_SUBAGENT_TASK_TOOLS: false
REQUIRE_OWNER_TO_COMPLETE: true
ENABLE_CYCLE_DETECTION: true
```

任务数据落盘到：

```text
.tasks/
├── index.json
├── events.jsonl
├── locks/
│   ├── index.lock
│   └── <task_id>.lock
└── tasks/
    └── task_000001.json
```

`index.json` 保存递增 ID 的高水位：

```json
{
  "version": 1,
  "next_id": 5,
  "tasks_dir": "tasks",
  "updated_at": "2026-07-07T22:00:00+08:00"
}
```

每个任务单独保存为 JSON：

```json
{
  "id": "task_000001",
  "subject": "设计 Task System 数据模型",
  "description": "设计任务状态、依赖关系、owner、metadata 等核心字段。",
  "status": "completed",
  "owner": "test_user",
  "active_form": "",
  "blocked_by": [],
  "blocks": ["task_000002", "task_000003"],
  "metadata": {
    "completion_summary": "测试完成：数据模型设计验证通过"
  },
  "created_at": "2026-07-07T22:08:56+08:00",
  "updated_at": "2026-07-07T22:09:34+08:00"
}
```

`events.jsonl` 追加记录任务事件：

```jsonl
{"time":"2026-07-07T22:08:56+08:00","event":"create","task_id":"task_000001","blocked_by":[]}
{"time":"2026-07-07T22:09:32+08:00","event":"claim","task_id":"task_000001","owner":"test_user"}
{"time":"2026-07-07T22:09:34+08:00","event":"complete","task_id":"task_000001","owner":"test_user","unblocked":["task_000002","task_000003"]}
```

## TaskManager

核心实现位于 `managers/task_manager.py`。

主要方法：

| 方法 | 作用 |
|---|---|
| `ensure_store()` | 创建 `.tasks/`、`tasks/`、`locks/`、`index.json`、`events.jsonl` |
| `create(...)` | 创建任务，校验依赖存在，写入 JSON |
| `get(task_id)` | 返回完整任务 JSON |
| `list_all(...)` | 返回任务看板摘要 |
| `update(task_id, **changes)` | 更新标题、描述、依赖、owner、状态、metadata |
| `can_start(task_id)` | 判断依赖是否全部完成 |
| `claim(task_id, owner)` | 在锁内认领任务，`pending -> in_progress` |
| `complete(task_id, owner, summary)` | 完成任务，返回刚刚解锁的下游任务 |
| `cancel(task_id, reason)` | 取消未完成任务 |
| `append_event(...)` | 追加事件日志 |

模块底部创建全局单例：

```python
TASK_MANAGER = TaskManager()
```

`base_handlers.py` 直接调用这个单例。

## ID 和锁

任务 ID 使用稳定递增格式：

```text
task_000001
task_000002
task_000003
```

`TaskManager._next_task_id()` 会用 `index.lock` 保护 `index.json`，避免多个进程同时创建任务时拿到同一个 ID。

认领和完成任务时使用任务级锁：

```text
.tasks/locks/task_000001.lock
```

锁内会重新读取任务，防止 TOCTOU：

1. 读取最新任务文件。
2. 检查状态是否仍是 `pending`。
3. 检查是否已有 `owner`。
4. 检查 `blocked_by` 是否全部完成。
5. 设置 `owner` 和 `status`。
6. 原子写回 JSON。

## 依赖和排序

`blocked_by` 是权威依赖来源。

例如 D 依赖 B 和 C：

```json
{
  "id": "task_000004",
  "blocked_by": ["task_000002", "task_000003"]
}
```

`blocks` 是反向索引，由 `_rebuild_blocks()` 自动重建。创建或更新依赖后，所有任务的 `blocks` 会重新计算。

`task_list` 默认排序：

1. `in_progress` 且有 owner。
2. `pending` 且依赖全部完成，显示为 `ready`。
3. `pending` 但依赖未完成，显示为 `blocked`。
4. `completed`。
5. `cancelled`。

列表输出示例：

```text
任务看板：4/4
- task_000002 [pending/ready] owner=- deps=task_000001 :: 实现 task_create/task_list/task_get 工具
- task_000003 [pending/ready] owner=- deps=task_000001 :: 实现 task_claim/task_complete 工具
- task_000004 [pending/blocked] owner=- deps=task_000002,task_000003 :: 编写 Task System 验收说明
- task_000001 [completed/closed] owner=test_user deps=- :: 设计 Task System 数据模型
```

## 状态机

当前实现的主流程：

```text
pending ── task_claim ──> in_progress ── task_complete ──> completed
   │
   └── task_cancel ──> cancelled
```

约束：

- `task_claim` 只允许认领 `pending` 任务。
- `task_claim` 前必须依赖全部完成。
- `task_complete` 只允许完成 `in_progress` 任务。
- `REQUIRE_OWNER_TO_COMPLETE=true` 时，如果传入 owner 且和当前 owner 不一致，会拒绝完成。
- `task_cancel` 不删除任务文件，只设置 `cancelled` 并清空 owner。

## 工具

当前项目把 Task System 工具直接定义在 `tools_configs/base_configs.py` 的 `TASK_TOOLS` 中，并合并进 `BASE_TOOLS`。

工具列表：

| 工具 | 作用 |
|---|---|
| `task_create` | 创建持久任务 |
| `task_list` | 查看任务看板摘要 |
| `task_get` | 查看完整任务 JSON |
| `task_update` | 更新任务字段和依赖 |
| `task_claim` | 认领 ready 的 pending 任务 |
| `task_complete` | 完成 in_progress 任务 |
| `task_cancel` | 取消未完成任务 |

这些 handler 直接写在 `handlers/base_handlers.py`：

```python
BASE_HANDLERS = {
    ...
    "task_create": handle_task_create,
    "task_list": handle_task_list,
    "task_get": handle_task_get,
    "task_update": handle_task_update,
    "task_claim": handle_task_claim,
    "task_complete": handle_task_complete,
    "task_cancel": handle_task_cancel,
}
```

### task_create

输入：

```json
{
  "subject": "设计 Task System 数据模型",
  "description": "设计任务状态、依赖关系、owner、metadata 等核心字段。",
  "blocked_by": [],
  "metadata": {}
}
```

说明：

- `subject` 必填。
- `blocked_by` 可选，但里面的任务 ID 必须已经存在。
- 如果依赖不存在，会返回工具错误给模型自纠。

### task_list

输入：

```json
{
  "include_completed": true,
  "owner": "test_user",
  "status": "pending"
}
```

所有字段都可选。默认包含 completed 和 cancelled。

### task_get

输入：

```json
{"task_id": "task_000004"}
```

返回完整 JSON，适合跨会话恢复任务说明。

### task_update

输入：

```json
{
  "task_id": "task_000004",
  "description": "新的完整任务说明",
  "blocked_by": ["task_000002", "task_000003"],
  "metadata": {"source": "manual-test"}
}
```

说明：

- `blocked_by` 是完整替换，不是增量追加。
- 更新依赖时会检查依赖任务存在。
- `ENABLE_CYCLE_DETECTION=true` 时会拒绝形成环的依赖。

### task_claim

输入：

```json
{
  "task_id": "task_000001",
  "owner": "test_user"
}
```

成功后状态变为 `in_progress`，owner 写入任务文件。

### task_complete

输入：

```json
{
  "task_id": "task_000001",
  "owner": "test_user",
  "summary": "测试完成：数据模型设计验证通过"
}
```

成功后状态变为 `completed`。如果有下游任务刚刚解锁，会在工具结果中返回。

### task_cancel

输入：

```json
{
  "task_id": "task_000001",
  "reason": "任务不再需要"
}
```

取消不会删除 JSON 文件。

## Prompt 接入

`prompts/task_prompts.py` 定义：

```python
TASK_SYSTEM_RULES = """持久任务规则：
- 大目标、跨会话目标、可并行目标或用户明确要求“拆任务/持久化/多人协作”时，优先使用 task_create 建立持久任务图。
- task 是跨会话工作看板；todo 是当前会话执行计划。不要把 todo 当作持久任务，也不要为简单一次性请求创建 task。
- 认领任务前先确认依赖已完成。被 blocked_by 阻塞的任务不能开始。
- 开始执行某个持久任务前调用 task_claim；完成后调用 task_complete。
- task_get 用于恢复完整任务说明；task_list 只用于看板摘要。
- 子 Agent 工具 task 和持久 Task System 不是同一个概念：task 工具是临时委托，task_create/task_claim/task_complete 管持久任务图。
"""
```

`prompts/prompt_sections.py` 把 `task_rules` 加入主 Agent section 顺序。

`managers/system_prompt_builder.py` 中：

```python
def _task_rules_prompt(agent_type: str) -> str:
    if not task_config.get("ENABLE_TASK_SYSTEM", True):
        return ""
    if agent_type == "subagent" and not task_config.get("ALLOW_SUBAGENT_TASK_TOOLS", False):
        return ""
    return TASK_SYSTEM_RULES
```

也就是说：

- 主 Agent 默认看到 Task System 规则。
- 子 Agent 默认不注入 Task System 规则，也没有 task 工具。
- 如果后续要让 teammate 自主认领任务，可以打开 `ALLOW_SUBAGENT_TASK_TOOLS` 并给子 Agent 工具列表接入 task 工具。

## 错误处理

Task System 的错误不走 s11 的模型调用重试。它们是工具执行错误，由 `handlers/dispatcher.py` 捕获：

```python
except Exception as exc:
    logger.exception("工具执行异常: %s", name)
    return f"Error while running {name}: {exc}"
```

这意味着：

- 工具参数错误不会让 Agent Loop 崩溃。
- 错误会作为 tool result 返回给模型。
- 模型下一轮可以根据错误自纠。

例如模型错误地使用不存在的依赖：

```json
{
  "subject": "实现工具",
  "blocked_by": ["task_pending"]
}
```

`TaskManager` 会拒绝：

```text
Error while running task_create: 依赖任务不存在: ['task_pending']
```

模型随后可以改用真实的 `task_000001` 重新创建。

`todo items 必须是列表` 这类错误也是同一类工具参数错误，不是 Task System 崩溃。

## Agent 启动时是否自动读取任务

当前实现**不会**在 `agent_loop.py` 启动时自动读取 `.tasks/`。

启动时发生的是：

1. `TASK_MANAGER = TaskManager()` 创建单例，只计算路径，不读文件。
2. `SessionStart` hook 初始化 Memory，不读取任务。
3. `build_main_system_prompt()` 注入 Task System 规则和工具摘要，不注入任务看板。

只有模型调用以下工具时才会读写 `.tasks/`：

- `task_list`
- `task_get`
- `task_create`
- `task_update`
- `task_claim`
- `task_complete`
- `task_cancel`

所以 Task System 当前提供的是“可恢复的持久任务图”，不是“启动后自动继续工作”的 autonomous loop。

自动扫描看板、空闲时认领任务、多个 teammate 自主工作，留给后续 Agent Teams / Autonomous Agents 阶段。

## 测试 Prompt

可以在 `agent_loop.py` 中临时使用下面的 `question` 测试：

```python
question = """
请测试持久 Task System，不要修改项目源代码，也不要调用 bash/read_file/write_file/edit_file。

请按顺序完成：
1. 使用 task_create 创建一个大目标拆分后的任务图：
   - 任务A：设计 Task System 数据模型
   - 任务B：实现 task_create/task_list/task_get 工具，依赖任务A
   - 任务C：实现 task_claim/task_complete 工具，依赖任务A
   - 任务D：编写 Task System 验收说明，依赖任务B和任务C
2. 调用 task_list 展示当前任务看板，确认依赖排序和 blocked/ready 状态。
3. 尝试认领任务D，预期应失败，因为依赖未完成。
4. 认领任务A，owner 使用 main-agent，然后完成任务A。
5. 再次调用 task_list，确认任务B和任务C已解锁。
6. 认领并完成任务B，owner 使用 main-agent。
7. 调用 task_get 查看任务D完整 JSON，确认它仍被任务C阻塞。
8. 最后用中文总结：哪些任务被创建、哪些任务被阻塞、哪些任务被解锁、Task System 是否能持久化到 .tasks/。
"""
```

观察重点：

- `.tasks/tasks/` 是否生成多个任务 JSON。
- `task_list` 是否按 ready / blocked / completed 排序。
- D 在 B/C 未完成时是否不能认领。
- A 完成后 B/C 是否解锁。
- B 完成后 D 是否仍被 C 阻塞。
- 重启后调用 `task_list` / `task_get` 是否能恢复任务状态。

## 验收

- 能创建持久任务文件。
- 能用 `blocked_by` 表达依赖。
- 依赖不存在时拒绝创建或更新。
- 依赖未完成时拒绝认领。
- 完成上游任务后能返回解锁的下游任务。
- `task_list` 能稳定展示 `ready` / `blocked` / `claimed` / `closed`。
- `task_get` 能返回完整 JSON。
- `task_update` 能更新依赖并拒绝环依赖。
- `task_claim` 使用文件锁，避免并发重复认领。
- Agent 重启后任务文件仍在，调用 `task_list` / `task_get` 能恢复进度。
