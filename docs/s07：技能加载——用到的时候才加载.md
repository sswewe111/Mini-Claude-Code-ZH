# s07：技能加载——用到的时候才加载

> 目标：让 Agent 知道有哪些技能可用，但不要把所有 `SKILL.md` 都塞进 system prompt。真正需要某个技能时，再通过 `load_skill` 工具把完整技能说明加载进当前上下文。

---

## 为什么需要 Skill Loading

当前项目的 `skills/` 目录已经包含多个较重的技能：

```text
skills/
├── docx/
├── frontend-design/
├── pdf/
├── pptx/
├── web-access/
└── xlsx/
```

这些技能的 `SKILL.md` 往往很长，还可能带有 `scripts/`、`reference.md`、`README.md` 等附加资源。如果启动时把所有内容直接放进 system prompt，会导致每一轮模型调用都携带大量无关上下文。

s07 的设计是两级加载：

| 层级 | 进入上下文的内容 | 进入时机 |
|---|---|---|
| 技能目录 | 技能名和描述摘要 | 每轮模型调用前由 system prompt 注入 |
| 技能正文 | 指定技能的完整 `SKILL.md` | 模型调用 `load_skill(name)` 后作为 tool result 注入 |

这样模型能先看到“有哪些技能”，但只有在任务真的需要时才加载完整技能说明。

## 当前实现结构

```text
configs/
└── skill_config.yml
state/
└── skill_state.py
managers/
├── skill_manager.py
└── system_prompt_builder.py
tools_configs/
├── base_configs.py
└── subagent_configs.py
handlers/
├── base_handlers.py
└── subagent_handlers.py
```

### `configs/skill_config.yml`

技能加载配置放在这里：

```yaml
SKILLS_ROOT: "skills"
SKILL_ENTRY_FILE: "SKILL.md"
INJECT_SKILL_CATALOG: true
MAX_SKILL_DESCRIPTION_CHARS: 180
MAX_SKILL_CONTENT_CHARS: 12000
ALLOW_SUBAGENT_LOAD_SKILL: true
```

含义：

- `SKILLS_ROOT`：技能根目录。
- `SKILL_ENTRY_FILE`：每个技能的入口文件。
- `INJECT_SKILL_CATALOG`：是否把技能目录摘要注入 system prompt。
- `MAX_SKILL_DESCRIPTION_CHARS`：限制目录摘要长度。
- `MAX_SKILL_CONTENT_CHARS`：限制 `load_skill` 返回内容长度。
- `ALLOW_SUBAGENT_LOAD_SKILL`：是否允许子 Agent 也使用 `load_skill`。

### `state/skill_state.py`

只保存技能元数据结构：

```python
@dataclass
class SkillManifest:
    name: str
    description: str
    path: Path
    entry_file: Path
```

它不保存完整 `SKILL.md` 正文。完整正文只会在 `load_skill` 被调用时读取并返回。

### `managers/skill_manager.py`

核心是 `SkillRegistry`：

- `scan_skills()`：扫描 `skills/*/SKILL.md`。
- `_parse_frontmatter()`：解析 `name`、`description` 等 frontmatter。
- `list_catalog()`：生成 system prompt 中使用的技能目录。
- `load_skill(name)`：按技能名读取完整 `SKILL.md`。

模块底部会创建全局注册表：

```python
SKILL_REGISTRY = SkillRegistry(skill_config)
SKILL_REGISTRY.scan_skills()
```

也就是说，程序启动并导入该模块时，会扫描一次技能目录。扫描阶段会读取 `SKILL.md` 来提取元数据，但注册表只保存 `SkillManifest`，不会把完整正文注入 prompt。

### `managers/system_prompt_builder.py`

主 Agent 和子 Agent 都通过这里构造 system prompt：

- `build_main_system_prompt(workdir)`
- `build_subagent_system_prompt(workdir)`

这两个函数都会追加同一份技能目录摘要：

```text
可用技能目录：
- docx: ...
- frontend-design: ...
- pdf: ...
- pptx: ...
- web-access: ...
- xlsx: ...

如果任务需要某个技能的完整说明，先调用 load_skill(name)。
```

注意：这里注入的是目录摘要，不是完整 `SKILL.md`。

## `load_skill` 工具

工具定义放在 `tools_configs/base_configs.py` 的 `SKILL_TOOLS` 中，并合并进 `BASE_TOOLS`。

工具参数只有一个：

```json
{
  "name": "docx"
}
```

不提供 `path` 参数，是为了避免模型传入任意路径读取项目敏感文件。

工具处理器放在 `handlers/base_handlers.py`：

```python
def handle_load_skill(args: dict) -> str:
    name = args.get("name", "")
    logger.info("加载技能: %s", name)
    return SKILL_REGISTRY.load_skill(name)
```

`load_skill("docx")` 返回内容包括：

- 技能名称。
- 入口文件路径。
- 完整或截断后的 `SKILL.md`。
- 如果内容过长，会提示已截断。

如果传入不存在的技能，例如 `load_skill("../.env")`，只会返回“技能不存在”和可用技能列表，不会读取 `.env`。

## 主 Agent 的调用链

主 Agent 的调用链如下：

```text
agent_loop.py
  ↓
build_main_system_prompt(workdir)
  ↓
system prompt 注入技能目录摘要
  ↓
模型判断需要某个技能
  ↓
调用 load_skill(name)
  ↓
dispatcher 分发到 handle_load_skill
  ↓
SkillRegistry.load_skill(name)
  ↓
完整 SKILL.md 作为 tool result 写回 messages
```

因此，主 Agent 每轮都能看到技能目录，但只有调用 `load_skill` 后，完整技能说明才进入对话历史。

---

## 子 Agent 的调用链

子 Agent 也具备技能加载能力。

相关配置：

- `configs/subagent_config.yml` 的 `ALLOWED_SUBAGENT_TOOLS` 包含 `load_skill`。
- `configs/skill_config.yml` 的 `ALLOW_SUBAGENT_LOAD_SKILL` 为 `true`。

子 Agent 的工具配置在 `tools_configs/subagent_configs.py`：

- 复用 `base_configs.py` 中的 `SKILL_TOOLS`。
- 仍然不暴露 `task`，避免递归创建子 Agent。
- 仍然不暴露 `todo`，避免污染父 Agent 的计划。

子 Agent 的工具映射在 `handlers/subagent_handlers.py`：

- 复用主 Agent 的 `handle_load_skill`。
- 调用时仍然经过 `PreToolUse` / `PostToolUse` Hook。
- 权限检查不会被绕过。

子 Agent 加载到的完整 `SKILL.md` 只进入子 Agent 自己的 `messages`。父 Agent 最后只收到子 Agent 的最终结果，不会自动继承子 Agent 加载过的完整技能正文。

---

## 权限与安全边界

`load_skill` 已加入 `configs/permission_config.yml` 的 allow 列表。

安全约束：

- 只能按注册表中的技能名加载。
- 不接收文件路径参数。
- 技能入口限制在 `skills/` 根目录下。
- 不自动读取 `scripts/`、`reference.md`、`README.md` 等附加资源。
- 附加资源需要模型根据 `SKILL.md` 指引，再通过已有工具按需读取。

这让 `load_skill` 更像“按名称打开一个已登记技能”，而不是“任意读文件工具”。

---

## 和前面阶段的关系

### 和 s04 Hook

`load_skill` 是普通工具调用，所以会经过：

- `PreToolUse`
- 权限检查 Hook
- `PostToolUse`
- 大输出提醒 Hook

### 和 s05 Todo

加载技能不是计划管理。复杂任务仍应先用 `todo` 写计划，再按计划调用 `load_skill`、`read_file`、`bash` 等工具。

### 和 s06 Subagent

父 Agent 可以自己加载技能，也可以把子任务交给子 Agent。子 Agent 如需专业能力，也可以自己调用 `load_skill`。

非 fork 模式下，子 Agent 从干净上下文开始，只看到技能目录摘要。fork 模式下，子 Agent 可以继承父上下文快照，但它加载技能后的完整正文仍然只保留在子 Agent 的上下文中，不自动写回父 Agent。

### 和 s08 Context Compact

s07 解决的是“不要提前塞进上下文”。s08 要解决的是“已经进入上下文的内容如何压缩或移除”。

## 本阶段完成标准

- 启动时能扫描 `skills/` 并生成技能目录。
- 主 Agent 的 system prompt 只包含技能摘要。
- 子 Agent 的 system prompt 也只包含技能摘要。
- 主 Agent 和子 Agent 都可以调用 `load_skill`。
- `load_skill` 返回完整或截断后的 `SKILL.md`。
- 技能正文不会在未调用工具前进入 messages。
- 不存在的技能名和路径穿越输入不会读取项目敏感文件。
