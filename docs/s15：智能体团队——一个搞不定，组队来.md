# s15：智能体团队——一个搞不定，组队来

s01 → ... → s13 → s14 → `s15` → s16 → s17 → s18 → s19

> “一个搞不定，组队来”——文件收件箱 + 队友线程。
>
> Harness 层：团队。它不替代 Agent Loop，而是在 Agent Loop 外侧增加 teammate 生命周期、消息总线和 Lead 收件箱注入。

---

## 问题

单个 Agent 处理大任务时，容易同时遇到三个限制：

- 上下文窗口有限，多个模块细节不能一直留在主上下文里。
- s06 的 `task` 子 Agent 是一次性调用，只返回最终结论，不适合持续协作。
- 有些任务天然可以并行，例如一个队友看配置，一个队友看工具实现，Lead 最后整合结果。

s15 的目标是把“调用一个临时子 Agent”扩展为“Lead + 多个 teammate”的协作形态。

## 配置

`configs/team_config.yml`：

```yaml
ENABLE_AGENT_TEAMS: true
TEAM_DIR: ".team"
TEAM_INBOX_DIR: ".team/inbox"
TEAM_ID: "default"
LEAD_AGENT_NAME: "lead"
MAX_TEAMMATES: 5
MAX_TEAMMATE_TURNS: 10
TEAM_INBOX_POLL_SECONDS: 1
MAX_INBOX_MESSAGES_PER_TURN: 20
TEAMMATE_RESULT_MAX_CHARS: 6000
```

这些配置由 `utils/config_handler.py` 加载为 `team_config`。其中：

- `TEAM_DIR` / `TEAM_INBOX_DIR` 决定文件收件箱位置。
- `MAX_TEAMMATES` 限制活跃 teammate 数量。
- `MAX_TEAMMATE_TURNS` 限制单个 teammate 的最大模型轮次。
- `MAX_INBOX_MESSAGES_PER_TURN` 限制每次读取 inbox 注入的消息数。

## 数据结构

`state/team_state.py` 提供两个核心结构。

`TeamMessage` 表示 inbox 中的一条消息：

```python
TeamMessage(
    id="team_msg_xxx",
    sender="alice",
    recipient="lead",
    type="result",
    content="任务结果...",
    created_at="2026-07-08T22:22:52+08:00",
    correlation_id=None,
    metadata={},
)
```

`TeamMemberRecord` 表示 `members.json` 中的一名队友：

```python
TeamMemberRecord(
    name="alice",
    role="team config reader",
    prompt="读取 configs/team_config.yml ...",
    status="running",
    thread_name="teammate-alice",
    turn_count=3,
)
```

## MessageBus：文件收件箱

`tools/message_bus.py` 实现 `MessageBus`：

- `ensure_store()`：创建 `.team/`、`.team/inbox/`、`.team/locks/`、`members.json`、`events.jsonl`。
- `send(sender, recipient, content, msg_type="message")`：向收件人的 `.jsonl` 文件追加一条 `TeamMessage`。
- `broadcast(sender, recipients, content)`：批量发送消息。
- `read_inbox(name, consume=True, limit=None)`：读取某个成员的 inbox。
- `append_event(event_type, payload)`：写入 `.team/events.jsonl`，用于排查创建、收发、读取事件。

当前代码不是 `read + unlink`，而是：

1. 读取 inbox 文件所有行。
2. 解析为 `TeamMessage`。
3. `consume=True` 时把已消费行从文件里移除，剩余消息写回原文件。

单进程多线程场景下，`MessageBus` 使用 `threading.RLock` 保护同一个 inbox 的读写。它不是跨进程文件锁；如果多个 Python 进程同时操作同一个 `.team/`，仍可能需要 s16/s18 再升级文件锁。

## TeamManager：团队后端

`managers/team_manager.py` 中的全局对象是：

```python
TEAM_MANAGER = TeamManager()
```

它负责：

- 初始化团队目录。
- 维护 `members.json`。
- 启动 teammate daemon 线程。
- 通过 `MessageBus` 给 Lead 或 teammate 发消息。
- 读取 Lead inbox 并格式化为 `<team_inbox>`。

关键方法：

- `start(client, model_id, hook_manager, workdir)`：在 `agent_loop.py` 启动时注入模型客户端和运行上下文。
- `spawn(name, role, prompt)`：创建 teammate 记录并启动线程。
- `list_all()`：列出成员状态。
- `send_message(to, content, msg_type="message")`：Lead 发送消息。
- `broadcast(content, recipients=None)`：Lead 广播给所有活跃 teammate。
- `check_inbox(agent="lead")`：读取 inbox。
- `drain_lead_inbox_for_prompt()`：给 hook 使用，读取 Lead inbox 并生成 `<team_inbox>`。

设计边界很明确：`TeamManager` 只负责调度和通信，不替 teammate 执行 bash/read/write。teammate 的执行发生在自己的 Agent 循环里。

## teammate 线程

`subagents/teammate_agent.py` 中的 `run_teammate_agent(...)` 是队友线程入口。

每个 teammate 有自己的：

- `LoopState(messages=...)`
- system prompt
- `RecoveryManager`
- 工具集
- 最大轮次限制

teammate 可用工具来自 `tools_configs/team_configs.py` 的 `TEAMMATE_AVAILABLE_TOOLS`：

- `bash`
- `read_file`
- `write_file`
- `edit_file`
- `team_send_message`
- `team_check_inbox`

teammate 不能使用 `spawn_teammate`，也不能继续创建新的 teammate。

执行流程：

1. Lead 调用 `spawn_teammate`。
2. `TeamManager.spawn()` 写入 `members.json`，状态为 `starting`。
3. 启动 daemon thread，线程入口为 `_run_teammate_thread()`。
4. teammate 进入 `run_teammate_agent()`，构造自己的初始任务消息。
5. 每轮先读取自己的 inbox，再调用模型。
6. 如果模型调用工具，就执行 teammate 工具并写回 tool result。
7. 如果模型不再调用工具，就把最终结果发送给 `lead`，状态改为 `idle`。
8. 如果超过 `MAX_TEAMMATE_TURNS`，发送 `error` 给 Lead，状态改为 `failed`。

当前 s15 的 teammate 是“任务完成后 idle”，不是长期驻留的 worker。也就是说，Lead 在 teammate 进入 `idle` 后再广播消息，消息会写入 `alice.jsonl` / `bob.jsonl`，但队友不会自动重新醒来消费。持续 idle loop、唤醒和 shutdown 协议属于 s16 的范围。

## Lead inbox 注入

`hooks/builtin_hooks.py` 中实现了两个团队 hook：

- `team_session_start_hook()`：会话启动时初始化 `.team/` 存储。
- `team_before_model_call_hook()`：主 Agent 每轮模型调用前读取 Lead inbox。

注册位置在 `hooks/hook_manager.py`：

```python
self.register("SessionStart", builtin_hooks.team_session_start_hook)
self.register("BeforeModelCall", builtin_hooks.team_before_model_call_hook)
```

注入后的消息格式：

```text
<team_inbox>
From alice [result]: configs/team_config.yml 中 Agent Teams 的关键配置...
From bob [result]: Agent Teams 暴露了 5 个团队工具...
</team_inbox>
```

Lead 不需要主动调用 `team_check_inbox` 才能看到队友结果；只要下一轮模型调用发生，hook 就会消费 `lead.jsonl` 并注入上下文。显式 `team_check_inbox` 仍保留，用于测试和手动查看。

## 工具接入

`tools_configs/team_configs.py` 定义 5 个 Lead 可见工具：

| 工具 | 作用 |
|---|---|
| `spawn_teammate` | 创建 teammate 并启动独立线程 |
| `team_send_message` | 向 Lead 或 teammate 发送消息 |
| `team_broadcast` | 向多个 teammate 广播消息 |
| `team_check_inbox` | 读取指定 agent 的 inbox |
| `team_list` | 查看成员状态 |

`tools_configs/base_configs.py` 把 `TEAM_TOOLS` 合并进 `BASE_TOOLS`。

`handlers/base_handlers.py` 新增对应 handler：

- `handle_spawn_teammate`
- `handle_team_send_message`
- `handle_team_broadcast`
- `handle_team_check_inbox`
- `handle_team_list`

`handlers/dispatcher.py` 对这些团队工具传入 `runtime_context`，这样 `spawn_teammate` 可以拿到当前 `client`、`model_id`、`hook_manager` 和 `workdir`。

权限方面，`configs/permission_config.yml` 已将团队工具加入 allow 列表：

```yaml
- spawn_teammate
- team_send_message
- team_broadcast
- team_check_inbox
- team_list
```

teammate 调用 `write_file` / `edit_file` 时仍走项目已有权限检查。

## Prompt 接入

`prompts/team_prompts.py` 定义两组规则：

- `TEAM_SYSTEM_RULES`：给 Lead，说明何时创建 teammate、如何广播、如何综合队友结果。
- `TEAMMATE_SYSTEM_RULES`：给 teammate，说明只处理被分配任务、通过 `team_send_message` 汇报、不要创建新 teammate。

`managers/system_prompt_builder.py` 增加：

- `_team_rules_prompt(agent_type)`
- `build_teammate_system_prompt(workdir, runtime_context=None)`

`prompts/prompt_sections.py` 增加 `TEAMMATE_SECTION_ORDER` 和 `TEAMMATE_IDENTITY_SECTION`。主 Agent 的 section 顺序里也加入了 `team_rules`。

## 示例流程

用户给 Lead 的测试问题：

```text
请测试 s15 Agent Teams 功能。不要修改项目源代码，不要调用 write_file/edit_file。

1. 调用 spawn_teammate 创建 alice，让她读取 configs/team_config.yml，总结关键配置，并把 result 发给 lead。
2. 调用 spawn_teammate 创建 bob，让他读取 tools_configs/team_configs.py 和 managers/team_manager.py，总结团队工具和 TeamManager 机制，并把 result 发给 lead。
3. 调用 team_list 查看成员状态。
4. 调用 team_broadcast 给所有 teammate 发送一条广播消息。
5. 等待后调用 team_check_inbox 查看 lead 的收件箱。
6. 最后总结是否证明“Lead 调度”和“teammate 执行”已解耦。
```

运行后会出现类似状态：

`members.json`：

```json
{
  "team_id": "default",
  "lead": "lead",
  "members": [
    {
      "name": "alice",
      "role": "team config reader",
      "status": "idle",
      "thread_name": "teammate-alice",
      "turn_count": 3
    },
    {
      "name": "bob",
      "role": "team tools reader",
      "status": "idle",
      "thread_name": "teammate-bob",
      "turn_count": 3
    }
  ]
}
```

`events.jsonl` 会记录：

```jsonl
{"type":"teammate_spawned","payload":{"name":"alice","status":"starting"}}
{"type":"message_sent","payload":{"sender":"alice","recipient":"lead","type":"result"}}
{"type":"inbox_read","payload":{"recipient":"lead","count":1,"consume":true}}
```

如果 `lead.jsonl` 当前为空，不代表 Lead 没收到消息。Lead inbox 是消费式读取：`team_before_model_call_hook` 或 `team_check_inbox` 读取后会把已消费消息从 `lead.jsonl` 移除。要查历史，应看 `.team/events.jsonl`。

如果 `alice.jsonl` / `bob.jsonl` 中还留着广播消息，说明 Lead 的广播已经投递成功，但 teammate 已完成任务并进入 `idle`，当前 s15 不会继续轮询自己的 inbox。

## 相对 s14 的变化

| 组件 | s14 | s15 |
|---|---|---|
| Agent 数量 | 单个主 Agent | 1 个 Lead + N 个 teammate 线程 |
| 调度对象 | cron event → Agent turn | Lead → teammate thread |
| 通信方式 | 无团队通信 | `.team/inbox/*.jsonl` 文件收件箱 |
| 新增 manager | `CronManager` | `TeamManager` |
| 新增工具 | `schedule_cron/list_crons/cancel_cron` | `spawn_teammate/team_send_message/team_broadcast/team_check_inbox/team_list` |
| 上下文注入 | 定时任务事件注入用户消息 | Lead inbox 注入 `<team_inbox>` |
| 解耦方式 | 定时触发与执行解耦 | Lead 调度与 teammate 执行解耦 |

## 验收标准

- `python -B -c "from tools_configs import BASE_TOOLS; from handlers.base_handlers import BASE_HANDLERS; ..."` 能导入团队工具。
- `BASE_TOOLS` 和 `BASE_HANDLERS` 中存在 5 个团队工具。
- 启动 Agent 后 `.team/members.json`、`.team/events.jsonl`、`.team/inbox/` 能自动创建。
- Lead 能调用 `spawn_teammate` 创建 teammate。
- teammate 能独立读取文件、调用工具，并通过 `team_send_message(..., msg_type="result")` 写入 `lead.jsonl`。
- Lead 能通过 hook 或 `team_check_inbox` 消费 `lead.jsonl`。
- `events.jsonl` 能看到 `teammate_spawned`、`message_sent`、`inbox_read`。
- 单个 teammate 异常不会杀掉 Lead 主循环；异常会以 `error` 消息发给 Lead。

## 当前边界

- teammate 完成初始任务后进入 `idle`，不会长期轮询 inbox。
- 当前没有 `shutdown_request` / `shutdown_approved` 等关闭协议。
- 当前没有权限冒泡协议；teammate 的工具权限仍走本地权限检查。
- 当前文件锁只覆盖单进程多线程，不覆盖多进程同时操作 `.team/`。
- 当前 Lead 自动消费 `lead.jsonl`，所以测试时看到 `lead.jsonl` 为空是正常现象，应结合 `events.jsonl` 判断历史消息。

s16 会在此基础上补团队协议：关闭握手、计划审批、消息类型约定和更完整的 teammate 生命周期。
