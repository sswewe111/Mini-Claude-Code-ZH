# s08：上下文压缩——上下文总会满，要有办法腾地方

s01 到 s07 已经让 Agent 具备了循环、工具、权限、Hook、计划管理、子 Agent 和技能加载能力。问题也随之出现：工具输出、技能正文、子任务结论和多轮对话会不断堆进 `messages`，上下文迟早会满。

s08 的目标是把 Context Compact 做成 Harness 层能力：压缩逻辑挂在 Hook 上，`agent_loop.py` 只负责触发生命周期事件，不直接关心“怎么压缩”。

## 核心设计

当前项目使用四类压缩：

- L1：大工具输出落盘。把超长工具结果写入 `.task_outputs/tool-results/`，上下文里只保留路径、原始长度和预览。
- L2：旧工具结果占位。保留最近几条完整工具结果，更旧的长结果替换为占位说明。
- L3：中间历史裁剪。消息条数过多时保留开头目标和最近上下文，中间替换为一条裁剪说明。
- L4：LLM 历史摘要。token 估算超过阈值，或模型主动请求 compact 时，调用模型生成中文摘要，并写 transcript。

另外还有 Reactive compact：当模型调用已经因为上下文过长失败时，执行一次应急 L4 摘要，然后重试。

## 文件结构

```text
configs/
└── compact_config.yml
state/
└── compact_state.py
managers/
└── compact_manager.py
prompts/
└── compact_prompts.py
hooks/
├── builtin_hooks.py
└── hook_manager.py
utils/
└── token_counter.py
tools_configs/
└── base_configs.py
handlers/
└── base_handlers.py
.task_outputs/
└── tool-results/
.transcripts/
```

各文件职责：

- `configs/compact_config.yml`：控制阈值、保留数量、输出目录、是否允许子 Agent 做 L4 摘要。
- `state/compact_state.py`：保存 compact 状态，例如上次摘要、transcript 路径、应急重试次数、手动 compact 请求。
- `managers/compact_manager.py`：实现所有压缩方法，是 Context Compact 的核心。
- `prompts/compact_prompts.py`：定义中文历史摘要提示词。
- `hooks/builtin_hooks.py`：把 compact 挂到 `BeforeModelCall` 和 `PostToolUse`。
- `utils/token_counter.py`：用字符数粗估 messages token。
- `tools_configs/base_configs.py`：定义 `compact` 工具描述。
- `handlers/base_handlers.py`：实现 `handle_compact`，登记手动压缩请求。

## 配置项

当前 `configs/compact_config.yml`：

```yaml
ENABLE_AUTO_COMPACT: true
MAX_MESSAGES_BEFORE_SNIP: 20
KEEP_HEAD_MESSAGES: 3
KEEP_TAIL_MESSAGES: 45
KEEP_RECENT_TOOL_RESULTS: 4
LARGE_TOOL_OUTPUT_CHARS: 12000
TOOL_OUTPUT_PREVIEW_CHARS: 1200
AUTO_COMPACT_TOKEN_THRESHOLD: 3000
MAX_REACTIVE_COMPACT_RETRIES: 1
MAX_SUMMARY_INPUT_CHARS: 30000
MAX_OUTPUT_TOKENS: 2048
ENABLE_SUBAGENT_LLM_COMPACT: false
TRANSCRIPT_DIR: ".transcripts"
TOOL_OUTPUT_DIR: ".task_outputs/tool-results"
```

说明：

- `LARGE_TOOL_OUTPUT_CHARS` 控制单条工具输出超过多少字符时落盘。
- `KEEP_RECENT_TOOL_RESULTS` 控制保留最近几条完整工具结果。
- `MAX_MESSAGES_BEFORE_SNIP` 控制消息数量超过多少时裁剪中间历史。
- `AUTO_COMPACT_TOKEN_THRESHOLD` 控制何时触发 L4 模型摘要。
- `MAX_OUTPUT_TOKENS` 会用于主模型调用和 compact 摘要调用。
- `ENABLE_SUBAGENT_LLM_COMPACT` 默认为 `false`，子 Agent 只做 L1/L2/L3，不主动触发 L4。

## 执行流程

### 1. 工具执行后：先处理单次大输出

工具 handler 执行完成后，`dispatcher` 触发 `PostToolUse`。

当前注册顺序是：

```text
PostToolUse -> log_post_tool_use
PostToolUse -> compact_post_tool_use_hook
```

执行过程：

1. `log_post_tool_use` 记录原始 `output_length`，方便排查哪个工具产生了大输出。
2. `compact_post_tool_use_hook` 调用 `CompactManager.persist_tool_output(...)`。
3. 如果输出长度超过 `LARGE_TOOL_OUTPUT_CHARS`：
   - 完整输出写入 `.task_outputs/tool-results/`。
   - Hook 返回 `HookResult(updated_output=...)`。
   - `dispatcher` 把压缩后的占位内容写入 `state.messages`。
4. 如果输出不大，原样写入 `state.messages`。

落盘后的占位格式类似：

```text
<persisted-tool-output path=".task_outputs/tool-results/xxx_bash.txt" original_chars="20000">
这里是预览内容
</persisted-tool-output>
[完整工具输出已落盘；如需完整内容，请使用 read_file 读取上述 path。]
```

### 2. 下一轮模型调用前：整理完整 messages

每轮调用模型前，`agent_loop.py` 触发 `BeforeModelCall`。

当前注册顺序是：

```text
BeforeModelCall -> todo_reminder_hook
BeforeModelCall -> compact_before_model_call_hook
BeforeModelCall -> log_before_model_call
```

执行过程：

1. `todo_reminder_hook` 先注入 Todo reminder。子 Agent 会跳过这个 Hook。
2. `compact_before_model_call_hook` 调用 `CompactManager.pre_model_compact(...)`。
3. `pre_model_compact` 内部按固定顺序执行：
   - `persist_large_tool_outputs(messages)`
   - `micro_compact_tool_results(messages)`
   - `snip_middle_messages(messages)`
   - 必要时 `compact_history(messages, client, model_id, focus)`
4. `log_before_model_call` 记录压缩后的 `messages` 数量。
5. `agent_loop.py` 执行 `normalize_messages(...)`。
6. 调用 `client.chat.completions.create(...)`，并传入 `max_tokens=_max_output_tokens()`。

### 3. L1 兜底：历史大工具结果落盘

`persist_large_tool_outputs(messages)` 会扫描历史里的 `role=tool` 消息。

触发条件：

- 消息角色是 `tool`。
- `content` 是字符串。
- 内容长度超过 `LARGE_TOOL_OUTPUT_CHARS`。
- 内容还不是 `<persisted-tool-output ...>` 占位。

作用：

- 处理没有在 `PostToolUse` 阶段被压缩的大输出。
- 处理旧代码运行后遗留在历史里的大工具结果。

### 4. L2：旧工具结果占位

`micro_compact_tool_results(messages)` 会找到所有 `role=tool` 消息。

触发条件：

- `role=tool` 消息数量超过 `KEEP_RECENT_TOOL_RESULTS`。
- 较旧工具结果内容长度大于 300 字符。

执行结果：

- 最近 N 条工具结果保持完整。
- 更旧的长工具结果替换为：

```text
[Earlier tool result compacted. 如需完整内容，请根据上下文重新读取对应文件或重新执行工具。]
```

### 5. L3：中间历史裁剪

`snip_middle_messages(messages)` 会在消息条数过多时裁剪中间历史。

触发条件：

- `len(messages) > MAX_MESSAGES_BEFORE_SNIP`。
- `KEEP_HEAD_MESSAGES + KEEP_TAIL_MESSAGES < len(messages)`。

执行结果：

- 保留开头 `KEEP_HEAD_MESSAGES` 条。
- 保留结尾 `KEEP_TAIL_MESSAGES` 条。
- 中间替换为一条 `role=user` 裁剪说明。
- 裁剪时保护 OpenAI 工具调用顺序：
  - 不让尾部从孤立的 `role=tool` 开始。
  - 不让头部以带 `tool_calls` 的 assistant 结束。
- 裁剪后调用 `normalize_messages(messages)`。

### 6. L4：LLM 历史摘要

`compact_history(messages, client, model_id, focus)` 是成本最高的一层，因为它会额外调用一次模型。

触发条件：

- `estimate_message_tokens(messages) >= AUTO_COMPACT_TOKEN_THRESHOLD`。
- 或上一轮模型调用了 `compact(focus=...)`，存在 `pending_manual_focus`。
- 并且当前上下文允许 L4。主 Agent 默认允许，子 Agent 默认不允许。

执行结果：

1. `write_transcript(messages)` 把完整历史写入 `.transcripts/transcript_*.jsonl`。
2. 使用 `COMPACT_HISTORY_PROMPT` 调用模型生成中文摘要。
3. 用一条摘要消息替换旧历史：

```text
[Context compacted]
完整 transcript: .transcripts/transcript_xxx.jsonl

这里是模型生成的中文摘要
```

摘要需要保留当前目标、已完成事项、未完成事项、关键文件、重要工具结果、用户约束、Todo 状态、已加载 Skill 和子 Agent 关键结论。

## 模型主动 compact

`compact` 是一个普通工具，定义在 `tools_configs/base_configs.py`，handler 在 `handlers/base_handlers.py`。

工具参数：

```json
{
  "focus": "压缩时重点保留什么信息"
}
```

当前实现不是“调用工具后立即改写 messages”，而是：

1. 模型调用 `compact(focus=...)`。
2. `handle_compact` 调用 `COMPACT_MANAGER.request_manual_compact(focus)`。
3. `request_manual_compact` 写入 `state.pending_manual_focus`。
4. 当前工具调用返回普通 tool result。
5. 下一轮 `BeforeModelCall` 中，`pre_model_compact` 发现 `pending_manual_focus`，执行 L4 `compact_history`。

这样设计是为了保持 OpenAI messages 的合法顺序：当前轮 assistant 的 `tool_calls` 必须对应当前轮的 `role=tool` 结果。如果在 tool handler 里立刻清空或重写 messages，很容易破坏这组对应关系。

## Reactive compact

`BeforeModelCall` 是主动预防，但仍可能发生模型 API 报上下文过长。

`agent_loop.py` 在调用：

```python
client.chat.completions.create(...)
```

时捕获异常。如果错误文本包含：

- `context length`
- `maximum context`
- `prompt too long`
- `too many tokens`
- `tokens exceed`
- `context_length_exceeded`
- `413`

就调用：

```python
COMPACT_MANAGER.reactive_compact(state.messages, client, model_id)
```

执行结果：

- 如果没有超过 `MAX_REACTIVE_COMPACT_RETRIES`，强制执行一次 `compact_history`。
- 然后回到下一轮循环重试模型调用。
- 如果超过重试次数，抛出原始错误，不无限循环。

Reactive compact 不是替代 `BeforeModelCall` 的常规压缩，而是失败后的兜底恢复。

## 子 Agent 中的 compact

子 Agent 也会触发 `BeforeModelCall`，所以它同样具备 Context Compact 能力。

当前默认行为：

- 子 Agent 会执行 L1：大工具结果落盘。
- 子 Agent 会执行 L2：旧工具结果占位。
- 子 Agent 会执行 L3：中间历史裁剪。
- 子 Agent 默认不会执行 L4：LLM 历史摘要。

原因是子 Agent 任务通常更短，默认不额外消耗一次模型调用。如果确实需要让子 Agent 也能做 L4，把 `configs/compact_config.yml` 中的配置改为：

```yaml
ENABLE_SUBAGENT_LLM_COMPACT: true
```

## 为什么压缩顺序是这样

执行顺序是：

```text
PostToolUse:
1. persist_tool_output

BeforeModelCall:
2. persist_large_tool_outputs
3. micro_compact_tool_results
4. snip_middle_messages
5. compact_history

Error:
6. reactive_compact
```

原因：

- `PostToolUse` 先处理本次大输出，避免大文本进入 `state.messages`。
- `BeforeModelCall` 再兜底扫描历史，处理遗漏的大工具结果。
- 旧工具结果占位要在中间历史裁剪前执行，先减少单条消息体积。
- 中间历史裁剪只处理消息数量，不理解内容，所以放在 LLM 摘要前；如果当前消息数还没有超过头尾保留数量之和，代码会跳过裁剪。
- LLM 摘要成本最高，只有 token 超阈值或手动请求时才执行。
- Reactive compact 只在模型调用失败后执行，不干扰正常循环。

## 如何测试

### 测试大输出落盘

让模型执行会产生长输出的命令，或调低配置：

```yaml
LARGE_TOOL_OUTPUT_CHARS: 500
```

运行：

```powershell
python .\agent_loop.py
```

观察日志是否出现：

```text
工具输出已落盘
工具输出已由 compact hook 落盘
```

并检查 `.task_outputs/tool-results/` 是否生成文件。

### 测试 L4 历史摘要

调低配置：

```yaml
AUTO_COMPACT_TOKEN_THRESHOLD: 1000
```

连续让 Agent 读取多个文件或执行多轮工具调用。

观察日志是否出现：

```text
触发 LLM compact
LLM compact 完成
```

并检查 `.transcripts/` 是否生成 transcript。

### 测试模型主动 compact

让模型明确执行：

```text
请调用 compact 工具，重点保留当前任务目标、已完成事项和剩余步骤。
```

预期行为：

1. 当前轮 `compact` 工具返回“已收到 compact 请求”。
2. 下一轮 `BeforeModelCall` 执行 L4。
3. `.transcripts/` 中出现新的 transcript。

## 验收标准

- 大工具输出不会无限进入 `messages`，而是写入 `.task_outputs/tool-results/`。
- 落盘占位里包含路径、原始长度和预览。
- 旧工具结果会被替换为占位说明。
- 消息过多时会裁剪中间历史，并保护 tool_calls / tool result 顺序。
- token 超阈值时会生成中文摘要，并写入 `.transcripts/`。
- `compact` 工具只登记请求，下一轮 `BeforeModelCall` 才真正压缩。
- 子 Agent 默认只做 L1/L2/L3，配置开启后才允许 L4。
- 模型上下文过长报错时，会执行一次 reactive compact，超过重试上限后抛出错误。
