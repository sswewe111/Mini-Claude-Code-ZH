# s17：自治智能体——自己看板，自己认领

[中文](README.md) | [English](README.en.md) | [日本語](README.ja.md)

s01 → ... → s15 → s16 → `s17` → [s18](./s18：%20Worktree%20Isolation%20—%20各干各的，互不干扰.md) → s19

> “自己看板，自己认领” 的目标是：Lead 只负责创建任务图和启动队友，空闲 teammate 自己扫描任务看板，发现 ready task 后自动认领、执行、完成，并把结果回传给 Lead。
>
> Harness 层：自治执行。它建立在 s12 任务系统、s15 团队消息总线、s16 团队协议之上，让团队成员从“等待 Lead 分配”升级为“空闲时自己找活干”。

## 当前实现范围

本项目的 s17 已落在当前 Python 工程中，核心文件如下：

| 文件 | 作用 |
|---|---|
| `managers/autonomous_manager.py` | 扫描 ready task、自动认领、写入自治事件、生成 `<claimed_task>` 指令 |
| `state/autonomous_state.py` | 定义一次自动认领的结果对象 `AutonomousClaimResult` |
| `managers/task_manager.py` | 提供 `ready_unowned_tasks()`、`claim()`、`complete()` 等任务状态能力 |
| `subagents/teammate_agent.py` | teammate 的工作循环、idle 扫描、收件箱处理、`task_complete` 后回传结果 |
| `tools_configs/team_configs.py` | 给 teammate 暴露 `task_list`、`task_get`、`task_complete` |
| `configs/team_config.yml` | 控制自治开关、扫描间隔、idle 超时、事件文件路径等配置 |
| `.team/autonomous_events.jsonl` | 运行时自治事件日志，记录 scan、claim_success、claim_failed、complete |

需要特别注意：当前实现没有暴露独立的 `task_claim` teammate 工具。自动认领由 `AutonomousManager.try_claim_next()` 在 teammate idle 阶段内部完成，底层调用 `TASK_MANAGER.claim()`。

## 解决的问题

s15 和 s16 之后，Lead 可以创建 teammate、收发团队消息、审批计划、请求关闭队友。但如果每个任务都要 Lead 手动分配，任务数量一多就会变成瓶颈。

s17 的设计把任务分配改成“看板驱动”：

1. Lead 创建一批持久任务，必要时设置 `blocked_by` 依赖。
2. Lead 启动一个或多个 teammate。
3. teammate 完成当前工作后进入 idle。
4. idle 阶段先检查自己的 inbox，再扫描 `.tasks` 中可开始且无人认领的任务。
5. 找到 ready task 后自动认领，把任务注入自己的上下文，回到模型工作循环。
6. 完成后调用 `task_complete`，系统自动记录完成事件并通知 Lead。

这样 Lead 不需要逐个 assign task，只需要维护任务图和观察结果。

## 配置项

当前配置在 `configs/team_config.yml`：

```yaml
ENABLE_AGENT_TEAMS: true
ENABLE_TEAM_PROTOCOLS: true
ENABLE_AUTONOMOUS_AGENTS: true

MAX_TEAMMATE_TURNS: 100

AUTONOMOUS_TASK_SCAN_ENABLED: true
AUTONOMOUS_IDLE_SCAN_SECONDS: 5
AUTONOMOUS_IDLE_TIMEOUT_SECONDS: 60
AUTONOMOUS_MAX_CLAIMS_PER_IDLE: 1
AUTONOMOUS_EVENT_FILE: ".team/autonomous_events.jsonl"
AUTONOMOUS_REQUIRE_TASK_METADATA_MATCH: false
AUTONOMOUS_ROLE_METADATA_KEY: "role"
AUTONOMOUS_NOTIFY_LEAD_ON_CLAIM: true
AUTONOMOUS_NOTIFY_LEAD_ON_COMPLETE: true
```

含义：

| 配置 | 当前值 | 说明 |
|---|---:|---|
| `ENABLE_AUTONOMOUS_AGENTS` | `true` | 总开关，关闭后 teammate 不会自动认领任务 |
| `AUTONOMOUS_TASK_SCAN_ENABLED` | `true` | 是否允许 idle teammate 扫描任务看板 |
| `AUTONOMOUS_IDLE_SCAN_SECONDS` | `5` | idle 阶段每隔多少秒扫描一次 inbox 和 ready task |
| `AUTONOMOUS_IDLE_TIMEOUT_SECONDS` | `60` | idle 阶段最长等待时间，超时后 teammate 停止 |
| `AUTONOMOUS_MAX_CLAIMS_PER_IDLE` | `1` | 每次 idle 扫描最多尝试认领几个任务 |
| `AUTONOMOUS_EVENT_FILE` | `.team/autonomous_events.jsonl` | 自治事件日志文件 |
| `AUTONOMOUS_REQUIRE_TASK_METADATA_MATCH` | `false` | 是否按任务 metadata 和 teammate role 做匹配过滤 |
| `AUTONOMOUS_NOTIFY_LEAD_ON_CLAIM` | `true` | 自动认领成功后是否通知 Lead |
| `AUTONOMOUS_NOTIFY_LEAD_ON_COMPLETE` | `true` | `task_complete` 成功后是否通知 Lead |
| `MAX_TEAMMATE_TURNS` | `100` | 单个 teammate 模型循环最多运行多少轮 |

`MAX_TEAMMATE_TURNS` 控制 teammate 的模型调用轮数；`AUTONOMOUS_IDLE_TIMEOUT_SECONDS` 控制 teammate 在 idle 阶段等待新任务的最长时间。这两个限制是不同维度。

## 任务何时算 ready

ready task 由 `TaskManager.ready_unowned_tasks()` 计算。一个任务必须同时满足：

1. `status == "pending"`。
2. `owner` 为空。
3. `blocked_by` 中的依赖任务都已经 `completed`。

底层逻辑复用 `TaskManager._can_start_task()`。因此，带依赖的任务不是永远不可认领，而是在上游任务完成后自动进入 ready 集合。

任务排序沿用 `TaskManager._sorted_tasks()`，因此多个 ready task 同时存在时，会按任务系统的既有排序取候选任务。

## 自动认领流程

核心入口是 `AutonomousManager.try_claim_next(teammate_name, role)`。

流程如下：

1. 检查 `ENABLE_AUTONOMOUS_AGENTS`。
2. 检查 `AUTONOMOUS_TASK_SCAN_ENABLED`。
3. 调用 `scan_ready_tasks()` 获取 ready 且未认领任务。
4. 如果启用了 `AUTONOMOUS_REQUIRE_TASK_METADATA_MATCH`，再用任务 metadata 过滤：
   - key 来自 `AUTONOMOUS_ROLE_METADATA_KEY`，默认是 `role`。
   - 任务没有该 metadata 时仍可匹配。
   - 任务 metadata 值需要包含在 teammate role 文本中。
5. 写入 `scan` 事件到 `.team/autonomous_events.jsonl`。
6. 对候选任务调用 `TASK_MANAGER.claim(task.id, owner=teammate_name)`。
7. 认领成功后：
   - 任务状态变为 `in_progress`。
   - 任务 owner 变为 teammate 名称。
   - `.tasks/events.jsonl` 记录 `claim`。
   - `.team/autonomous_events.jsonl` 记录 `claim_success`。
   - 返回 `AutonomousClaimResult(claimed=True, task=...)`。
8. 认领失败时记录 `claim_failed`。

简化后的伪代码：

```python
def try_claim_next(teammate_name, role):
    if not ENABLE_AUTONOMOUS_AGENTS:
        return not_claimed
    if not AUTONOMOUS_TASK_SCAN_ENABLED:
        return not_claimed

    tasks = scan_ready_tasks(teammate_name, role)
    for task in tasks[:AUTONOMOUS_MAX_CLAIMS_PER_IDLE]:
        result = TASK_MANAGER.claim(task.id, owner=teammate_name)
        if result.startswith("已认领"):
            append_event("claim_success", teammate_name, {"task_id": task.id})
            return AutonomousClaimResult(True, result, load_task(task.id))

        append_event("claim_failed", teammate_name, {"task_id": task.id, "message": result})

    return AutonomousClaimResult(False, "扫描到任务，但认领失败。")
```

## teammate 生命周期

s17 没有单独新建一个 Agent 类型，而是扩展了 s15/s16 的 teammate 线程。

teammate 主循环在 `subagents/teammate_agent.py`：

1. 每轮开始先读取自己的 inbox。
2. 如果收到 `shutdown_request` 等协议消息，交给 s16 的协议处理。
3. 调用模型，让 teammate 使用工具完成工作。
4. 如果模型返回了工具调用，就执行工具并继续下一轮。
5. 如果模型没有工具调用，说明这一轮工作结束：
   - 有文本结果时先用 `team_send_message` 发给 Lead。
   - 进入 `_idle_until_message()`。
6. idle 阶段：
   - 把成员状态写为 `idle`。
   - 可选发送 `idle_notification` 给 Lead。
   - 循环读取 inbox。
   - 没有消息时调用 `AUTONOMOUS_MANAGER.try_claim_next()` 扫描并认领任务。
   - 如果认领成功，把 `<claimed_task>` 注入上下文，状态改回 `running`，回到模型循环。
   - 如果超过 `AUTONOMOUS_IDLE_TIMEOUT_SECONDS`，状态改为 `stopped` 并退出。

这一点很关键：自动认领只在 teammate 进入 idle 后触发。如果模型一直在工具循环中，没有产生“无工具调用”的结束回合，就不会进入 idle，也就不会触发自动扫描。

## `<claimed_task>` 注入

认领成功后，`AutonomousManager.format_claimed_task()` 会生成一段结构化指令注入 teammate 的上下文：

```xml
<claimed_task>
id: task_000001
subject: ...
owner: auto_bob
blocked_by: -
description:
...

你已成功认领该任务。完成后必须调用 task_complete(task_id="task_000001", owner="auto_bob", summary="...")，然后用 team_send_message 向 lead 汇报。
</claimed_task>
```

这段内容告诉 teammate：

- 当前任务已经被系统认领。
- owner 是 teammate 自己。
- 完成后必须调用 `task_complete`。
- 完成后还需要向 Lead 汇报。

## teammate 可用工具

s17 给 teammate 增加了任务看板工具：

| 工具 | 作用 |
|---|---|
| `task_list` | 查看任务看板摘要，可按 owner 或 status 过滤 |
| `task_get` | 查看某个任务完整信息 |
| `task_complete` | 完成当前 teammate 已认领的任务 |

其中 `task_complete` 的实现会调用 `TASK_MANAGER.complete()`。只有 `in_progress` 的任务可以完成；如果配置 `REQUIRE_OWNER_TO_COMPLETE` 开启，则 owner 需要匹配。

当 `task_complete` 成功，并且 `AUTONOMOUS_NOTIFY_LEAD_ON_COMPLETE=true` 时，系统还会：

1. 给 Lead 发送 `msg_type="result"` 的团队消息。
2. 在 `.team/autonomous_events.jsonl` 中写入 `complete` 事件。
3. 在 `.tasks/events.jsonl` 中写入任务系统自己的 `complete` 事件，并记录刚刚解锁的下游任务。

## 事件与运行态文件

s17 相关运行态主要分布在两个目录：

| 文件 | 内容 |
|---|---|
| `.tasks/*.json` | 每个持久任务的状态、owner、依赖、metadata |
| `.tasks/index.json` | 任务系统索引 |
| `.tasks/events.jsonl` | 任务系统事件，如 create、claim、complete |
| `.team/members.json` | teammate 成员、状态、轮数、错误信息 |
| `.team/events.jsonl` | 团队消息事件，如 message_sent、inbox_read |
| `.team/autonomous_events.jsonl` | 自治事件，如 scan、claim_success、claim_failed、complete |
| `.team/inbox/*.jsonl` | 成员收件箱；只有收到消息的成员才一定会生成对应文件 |

`.team/autonomous_events.jsonl` 的典型记录：

```json
{"type":"scan","teammate":"auto_bob","created_at":"...","payload":{"ready_count":2}}
{"type":"claim_success","teammate":"auto_bob","created_at":"...","payload":{"task_id":"task_000005","message":"已认领..."}}
{"type":"complete","teammate":"auto_bob","created_at":"...","payload":{"task_id":"task_000005","message":"已完成..."}}
```

排查 s17 时不要只看 teammate 的 inbox。inbox 是队列，读取消费后可能为空；历史消息应优先看 `.team/events.jsonl`，自治链路应看 `.team/autonomous_events.jsonl` 和 `.tasks/events.jsonl`。

## 示例流程

下面是一个典型的 s17 运行流程：

1. Lead 调用 `task_create` 创建三个任务：
   - `task_000001`：读取配置。
   - `task_000002`：总结自治管理器，`blocked_by=["task_000001"]`。
   - `task_000003`：总结 teammate 执行流程，`blocked_by=["task_000001"]`。
2. Lead 调用 `spawn_teammate` 启动 `auto_alice` 和 `auto_bob`。
3. teammate 完成初始说明后进入 idle。
4. `auto_bob` idle 扫描 ready task，发现 `task_000001`。
5. `AUTONOMOUS_MANAGER.try_claim_next("auto_bob")` 调用 `TASK_MANAGER.claim()`：
   - `task_000001.status = "in_progress"`。
   - `task_000001.owner = "auto_bob"`。
   - 写入 `claim_success`。
6. 系统把 `<claimed_task>` 注入 `auto_bob` 上下文。
7. `auto_bob` 根据任务读取文件、总结结果。
8. `auto_bob` 调用 `task_complete(task_id="task_000001", owner="auto_bob", summary="...")`。
9. 任务系统把 `task_000001` 标记为 `completed`，并发现 `task_000002`、`task_000003` 依赖已满足。
10. `auto_bob` 或其他 idle teammate 下一轮扫描时，可以认领刚解锁的任务。
11. Lead 通过 `team_check_inbox` 或模型前 hook 读取结果消息。

这条链路体现的是“调度权交给任务看板”：Lead 不需要知道每个 teammate 什么时候空闲，也不需要逐个分配下游任务。

## 测试建议

可以用类似下面的问题测试 s17：

```text
请创建一个 s17 Autonomous Agents 测试任务图：
1. 创建任务 A：读取 configs/team_config.yml，总结自治相关配置。
2. 创建任务 B：读取 managers/autonomous_manager.py，总结自动扫描和认领流程，依赖 A。
3. 创建任务 C：读取 subagents/teammate_agent.py，总结 teammate idle 后如何自动认领任务，依赖 A。
然后启动 auto_alice 和 auto_bob 两个 teammate。
要求 teammate 不要修改任何文件，只读取和总结；让它们进入 idle 后自己认领 ready task。
最后检查 team inbox、.team/autonomous_events.jsonl、.tasks/events.jsonl，并总结是否完成 scan -> claim_success -> task_complete -> 解锁下游任务的闭环。
```

观察点：

- `.team/members.json` 中 teammate 是否从 `running` 进入 `idle`，再回到 `running`。
- `.team/autonomous_events.jsonl` 是否出现 `scan`、`claim_success`、`complete`。
- `.tasks/*.json` 中任务是否从 `pending` 变成 `in_progress`，最后变成 `completed`。
- 下游任务是否在依赖完成后变成 ready。
- Lead 是否收到 teammate 的 `result` 消息。
- 是否触发 `MAX_TEAMMATE_TURNS` 或 `AUTONOMOUS_IDLE_TIMEOUT_SECONDS` 限制。

## 常见问题

### 为什么 teammate 没有自动认领？

优先检查：

1. `ENABLE_AUTONOMOUS_AGENTS` 是否为 `true`。
2. `AUTONOMOUS_TASK_SCAN_ENABLED` 是否为 `true`。
3. teammate 是否真的进入了 idle。自动扫描只在 `_idle_until_message()` 中触发。
4. 是否存在 ready task：任务必须是 `pending`、无 owner、依赖全部完成。
5. `MAX_TEAMMATE_TURNS` 是否过低，导致 teammate 还没进入 idle 就失败。
6. 如果启用了 `AUTONOMOUS_REQUIRE_TASK_METADATA_MATCH`，任务 metadata 是否和 teammate role 匹配。

### 为什么没有 `auto_alice.jsonl`？

`.team/inbox/<name>.jsonl` 只有在有消息发给该成员时才会创建。成员注册在 `.team/members.json`，不代表一定有对应 inbox 文件。历史消息看 `.team/events.jsonl`。

### 为什么 `auto_bob.jsonl` 是空的？

inbox 是队列。teammate 读取时默认 `consume=True`，读完会清空收件箱。空文件不代表没有收过消息，应查看 `.team/events.jsonl` 中的 `message_sent` 和 `inbox_read`。

### 为什么任务完成后没有继续做下一个？

可能原因：

- teammate 已达到 `MAX_TEAMMATE_TURNS`。
- teammate idle 超过 `AUTONOMOUS_IDLE_TIMEOUT_SECONDS` 后停止。
- 没有新的 ready task。
- 模型没有正确调用 `task_complete`，任务仍停在 `in_progress`。

## 相对 s16 的变化

| 维度 | s16 Team Protocols | s17 Autonomous Agents |
|---|---|---|
| 工作来源 | Lead 通过消息或 prompt 指派 | teammate idle 后自己扫描任务看板 |
| 任务分配 | 人工分配为主 | 自动认领 ready 且无 owner 的任务 |
| 依赖处理 | Lead 需要关注依赖状态 | `ready_unowned_tasks()` 自动过滤未满足依赖 |
| teammate idle | 等待消息或 shutdown | 等待消息，同时扫描任务 |
| 完成动作 | 发消息汇报即可 | 必须调用 `task_complete` 完成任务状态闭环 |
| 事件记录 | `.team/events.jsonl`、`.team/requests.json` | 增加 `.team/autonomous_events.jsonl` |
| 工具能力 | 团队消息和协议工具 | 增加 `task_list`、`task_get`、`task_complete` |
