# s16：团队协议——队友之间要有约定

s01 → ... → s14 → s15 → `s16` → s17 → s18 → s19

> “队友之间要有约定”——用 request-response 协议让 Lead 和 teammate 可以协商、审批和体面关闭。
>
> Harness 层：协议。它建立在 s15 的文件 inbox 和 teammate 线程之上，为团队消息增加结构化类型、`request_id` 和状态追踪。

---

## 问题

s15 已经实现了 Agent Teams：Lead 可以创建 teammate，双方通过 `.team/inbox/*.jsonl` 发送消息。但 s15 的消息还是松散文本，带来三个问题：

- Lead 无法可靠判断一条回复对应哪一个请求。
- teammate 完成初始任务后进入 `idle` 状态，但早期实现会退出循环，无法继续消费后续消息。
- 关闭 teammate、审批高风险计划这类动作需要握手，而不能靠普通聊天文本。

s16 的目标是把普通团队消息升级为 request-response 协议。

## 配置

`configs/team_config.yml` 增加协议配置：

```yaml
ENABLE_TEAM_PROTOCOLS: true
TEAM_REQUEST_FILE: ".team/requests.json"
TEAM_REQUEST_ID_PREFIX: "team_req_"
TEAM_REQUEST_TIMEOUT_SECONDS: 300
TEAM_IDLE_POLL_SECONDS: 1
TEAM_IDLE_MAX_SECONDS: 0
TEAM_IDLE_NOTIFY_LEAD: true
PLAN_APPROVAL_REQUIRED: true
PLAN_APPROVAL_MAX_CHARS: 6000
```

说明：

- `TEAM_REQUEST_FILE`：协议状态文件位置。
- `TEAM_REQUEST_ID_PREFIX`：生成 request id 的前缀，例如 `team_req_000001`。
- `TEAM_IDLE_POLL_SECONDS`：teammate idle 后轮询 inbox 的间隔。
- `TEAM_IDLE_MAX_SECONDS=0`：不因 idle 超时自动退出。
- `TEAM_IDLE_NOTIFY_LEAD=true`：teammate 进入 idle 后向 Lead 发 `idle_notification`。

## 协议状态

`state/team_protocol_state.py` 定义 `ProtocolRequestRecord`：

```python
@dataclass
class ProtocolRequestRecord:
    request_id: str
    protocol: str              # shutdown / plan_approval
    sender: str
    recipient: str
    status: str                # pending / approved / rejected
    request_type: str
    response_type: str = ""
    payload: dict = field(default_factory=dict)
    response: dict = field(default_factory=dict)
    created_at: str = ""
    updated_at: str = ""
    resolved_at: str | None = None
```

`.team/requests.json` 会保存请求状态，例如：

```json
{
  "updated_at": "2026-07-08T23:10:00+08:00",
  "requests": [
    {
      "request_id": "team_req_000001",
      "protocol": "shutdown",
      "sender": "lead",
      "recipient": "protocol_alice",
      "status": "approved",
      "request_type": "shutdown_request",
      "response_type": "shutdown_approved"
    }
  ]
}
```

## TeamProtocolManager

`managers/team_protocol_manager.py` 中的全局对象是：

```python
TEAM_PROTOCOL_MANAGER = TeamProtocolManager()
```

它负责请求状态机：

- `ensure_store()`：创建 `.team/requests.json`。
- `create_request(...)`：生成 `team_req_000001` 形式的 request id，写入 pending 状态。
- `match_response(message)`：根据 `message.metadata["request_id"]` 匹配响应。
- `resolve_request(request_id, response_type, response)`：Lead 主动审批计划时直接更新状态。
- `get_request(request_id)`：读取单个请求。
- `format_requests(...)`：格式化请求状态，供 `team_protocol_status` 返回。

响应类型映射：

```python
EXPECTED_RESPONSES = {
    "shutdown": {"shutdown_approved", "shutdown_rejected"},
    "plan_approval": {"plan_approval_approved", "plan_approval_rejected"},
}
```

状态转换：

- `shutdown_approved` / `plan_approval_approved` → `approved`
- `shutdown_rejected` / `plan_approval_rejected` → `rejected`

如果 request id 不存在、协议类型不匹配、请求已经不是 pending，状态不会被错误更新。

## 消息类型

s16 当前实现支持这些协议消息：

| 类型 | 方向 | 作用 |
|---|---|---|
| `idle_notification` | teammate → Lead | 队友进入 idle，等待新消息或关闭请求 |
| `shutdown_request` | Lead → teammate | 请求 teammate 体面关闭 |
| `shutdown_approved` | teammate → Lead | teammate 同意关闭 |
| `shutdown_rejected` | teammate → Lead | 预留：teammate 拒绝关闭 |
| `teammate_terminated` | teammate → Lead | teammate 线程已退出 |
| `plan_approval_request` | teammate → Lead | teammate 提交计划审批 |
| `plan_approval_approved` | Lead → teammate | Lead 批准计划 |
| `plan_approval_rejected` | Lead → teammate | Lead 拒绝计划 |

协议消息通过 `TeamMessage.metadata` 携带 request id：

```json
{
  "request_id": "team_req_000002",
  "protocol": "plan_approval"
}
```

## TeamManager 接入

`managers/team_manager.py` 在 s16 中承担协议门面：

- `request_shutdown(name, reason="")`
  - 创建 `shutdown` 请求。
  - 向 teammate inbox 写入 `shutdown_request`。
  - 返回 `request_id`。

- `submit_plan(sender, plan, risk_level="", target_files=None)`
  - teammate 调用。
  - 创建 `plan_approval` 请求。
  - 向 Lead inbox 写入 `plan_approval_request`。

- `review_plan(request_id, approve, feedback="")`
  - Lead 调用。
  - 更新 `.team/requests.json`。
  - 向 teammate inbox 写入 `plan_approval_approved` 或 `plan_approval_rejected`。

- `protocol_status(request_id="", include_resolved=True)`
  - 查看协议请求状态。

- `route_protocol_messages(messages, recipient)`
  - Lead 消费 inbox 后，先路由协议响应。
  - `shutdown_approved`、`shutdown_rejected`、`plan_approval_approved`、`plan_approval_rejected` 会调用 `TeamProtocolManager.match_response()`。
  - `teammate_terminated` 会把成员状态更新为 `stopped`。

Lead inbox 的行为仍是消费式读取。`team_check_inbox` 或 `team_before_model_call_hook` 读走消息后，`lead.jsonl` 可能变空；历史和状态应看 `.team/events.jsonl` 与 `.team/requests.json`。

## teammate idle loop

`subagents/teammate_agent.py` 已改为：

```text
running -> idle -> running -> idle -> stopped
```

关键行为：

1. teammate 完成一轮模型任务后，如果没有更多 tool call：
   - 通过 `team_send_message` 或自动 result 向 Lead 汇报。
   - 调用 `_idle_until_message(...)` 进入 idle。
2. idle 时每 `TEAM_IDLE_POLL_SECONDS` 读取自己的 inbox。
3. 收到普通消息时，把消息注入 `<team_inbox>`，回到 running。
4. 收到 `shutdown_request` 时，调用 `handle_teammate_protocol_messages(...)`：
   - 发送 `shutdown_approved` 给 Lead。
   - 发送 `teammate_terminated` 给 Lead。
   - 更新成员状态为 `stopped`。
   - 线程退出。
5. 收到 `plan_approval_approved` / `plan_approval_rejected` 时，把审批结果作为普通上下文注入，让模型继续判断下一步。

这解决了 s15 中“队友 idle 后不会消费后续消息”的问题。

## 计划审批流程

计划审批当前实现的是“消息协议 + prompt 约束”，不是强制工具门控。

teammate 调用：

```text
team_submit_plan(
  plan="阅读 managers/team_manager.py 和 managers/team_protocol_manager.py，总结 request_id 匹配机制",
  risk_level="medium",
  target_files=["managers/team_manager.py", "managers/team_protocol_manager.py"]
)
```

系统行为：

1. `TeamManager.submit_plan()` 创建 `plan_approval` request。
2. `.team/requests.json` 写入 pending。
3. `lead.jsonl` 收到 `plan_approval_request`。
4. Lead 看到请求后调用：

```text
team_review_plan(
  request_id="team_req_000002",
  approve=true,
  feedback="计划批准，只允许读取文件和总结。"
)
```

5. `.team/requests.json` 中该请求变为 `approved`。
6. teammate inbox 收到 `plan_approval_approved`，idle loop 读取后继续执行。

如果拒绝，则发送 `plan_approval_rejected`，teammate 收到反馈后应调整计划或停止。

## 关闭握手流程

Lead 调用：

```text
team_request_shutdown(
  name="protocol_alice",
  reason="s16 shutdown protocol test finished"
)
```

系统行为：

1. `TeamManager.request_shutdown()` 创建 `shutdown` request。
2. `.team/requests.json` 写入 pending。
3. `.team/inbox/protocol_alice.jsonl` 收到 `shutdown_request`。
4. protocol_alice 的 idle loop 读取 inbox。
5. `subagents/team_protocols.py` 处理 `shutdown_request`。
6. protocol_alice 向 Lead 发送 `shutdown_approved`。
7. protocol_alice 再发送 `teammate_terminated`。
8. Lead 消费 inbox 后，`TeamProtocolManager.match_response()` 把 request 状态改成 `approved`。
9. `TeamManager.route_protocol_messages()` 把成员状态改成 `stopped`。

## 示例测试问题

可以在 `agent_loop.py` 的固定 `question` 中使用：

```text
请测试 s16 Team Protocols 功能。不要修改项目源代码，不要调用 write_file/edit_file。使用全新的 teammate 名称 protocol_alice 和 protocol_bob。

1. 创建 protocol_alice，让她读取 configs/team_config.yml 并汇报，然后 idle。
2. 等待后调用 team_request_shutdown(name="protocol_alice")，记录 request_id。
3. 调用 team_protocol_status 查看 shutdown 请求是否 approved。
4. 创建 protocol_bob，让他先调用 team_submit_plan 提交计划审批请求。

6. 调用 team_review_plan 批准该计划。
7. 等待 protocol_bob 收到审批后继续执行并返回 result。
8. 最后调用 team_protocol_status 展示所有请求状态。
```

## 相对 s15 的变化

| 组件 | s15 | s16 |
|---|---|---|
| 消息 | `message/result/error` | 增加协议消息类型 |
| 请求状态 | 无 | `.team/requests.json` |
| 管理器 | `TeamManager` | `TeamManager + TeamProtocolManager` |
| teammate 生命周期 | 完成后 idle/退出 | idle loop 继续轮询 inbox |
| 关闭 | 无握手 | `shutdown_request -> shutdown_approved -> teammate_terminated` |
| 计划审批 | 无 | `plan_approval_request -> approved/rejected` |
| 调试工具 | `team_list/team_check_inbox` | 增加 `team_protocol_status` |

## 当前边界

- `shutdown_rejected` 已纳入匹配表，但当前 teammate 侧默认收到 `shutdown_request` 后直接批准。
- 计划审批目前依赖 prompt 约束，未在 `PreToolUse` 强制阻断未审批的 `bash/write_file/edit_file`。
- `TEAM_REQUEST_TIMEOUT_SECONDS` 已配置，但当前 `TeamProtocolManager` 没有定时过期扫描。
- `.team/requests.json` 是本地文件状态，适合单进程教学场景；多进程并发仍需要后续文件锁增强。

---

## 验收标准

- `team_request_shutdown` 能生成 `team_req_000001` 形式的 request id。
- `.team/requests.json` 能看到 shutdown / plan_approval 请求状态。
- teammate idle 后能继续读取自己的 inbox。
- teammate 收到 `shutdown_request` 后能发送 `shutdown_approved` 和 `teammate_terminated`。
- Lead 消费 inbox 后，shutdown 请求从 `pending` 变为 `approved`。
- teammate 能调用 `team_submit_plan` 发起 `plan_approval_request`。
- Lead 能调用 `team_review_plan` 发送 approved/rejected 响应。
- `team_protocol_status` 能展示 pending/approved/rejected 请求。
